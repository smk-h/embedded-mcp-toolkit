## 一、 什么时候用？

不是所有代码都需要状态机。当出现以下信号时，说明状态机比 if-else 更适合：

### 1. 大量 if-else / switch 嵌套

代码在不同条件下跳转不同分支，且分支之间有先后依赖，难以一眼看清完整路径。

```typescript
// 反例 —— 需要状态机来整理
if (banner.includes("login:")) { /* 分支A */ }
else {
  wait();
  const output = read();
  if (output.includes("Password:")) {
    if (output.includes("login:")) { /* 分支B */ }
    else {
      write(probe);
      wait();
      const out2 = read();
      if (out2.includes("incorrect")) { /* 分支C */ }
      else if (out2.includes("__SH_STATUS_PROBE__")) { /* 分支D */ }
      else { /* 分支E */ }
    }
  }
  else if (output.includes("__SH_STATUS_PROBE__")) { /* 分支F */ }
  else { /* 分支G */ }
}
```

### 2. 输入的含义随阶段变化

同一个 `"Password:"` 输出，在不同阶段含义完全不同：

- 探测阶段出现 → 终端在等密码，可能是探测命令被当密码吞掉了
- 登录阶段出现 → 发完用户名后的正常响应，下一步应该发密码

这种“上下文敏感”的输入，最适合用状态来区分。

### 3. 存在重试 / 回退逻辑

像探测命令被当密码吃掉后需要“二次探测”，同一套判断逻辑可能跑两次。如果当前已处于某个探测阶段，输出又模糊不清，需要再加一轮——这就是状态机中的“自循环”。

### 4. 判断方法

画一张简单的表：

| 当前阶段 | 输入 | 下一阶段 |
|---|---|---|
| 未知 | banner 含 login: | 等待用户名 |
| 未知 | 探测含 probe | 已登录 |
| 未知 | 探测含 Password: + login: | 等待用户名 |
| 未知 | 探测含 Password:（无 login:） | 二次探测中 |
| 二次探测中 | 输出含 probe | 已登录 |
| 二次探测中 | 输出含 incorrect / Password: / login: | 等待用户名 |
| 二次探测中 | 异常 | 错误 |

这张表就是状态机的核心——**每个格子都不需要再套一层 if-else**，逻辑变得扁平。

## 二、 状态机设计

### 1. 枚举所有状态

先把“当前是什么情况”列出来。每个状态代表系统在某个时间点的唯一情况。

```typescript
export enum UserLoginStatus {
  UNKNOWN            = "unknown",              // 未知：刚连接，还未判断
  NO_LOGIN_REQUIRED  = "no_login_required",    // 已登录：可以直接操作设备
  WAITING_USERNAME   = "waiting_username",     // 等待用户名：终端显示 login:
  WAITING_PASSWORD   = "waiting_password",     // 等待密码：终端显示 Password:
  LOGGED_IN          = "logged_in",            // 登录成功
  WRONG_KEY          = "wrong_key",            // 密码错误
  ERROR              = "error",                // 不可恢复的错误
}
```

【**注意**】状态是**语义标签**，不是过程步骤。不要把“正在发探测”“正在验证密码”这种过程步骤放进状态枚举里——这些是转换中的动作，用内部变量（如 `_probeCount`）跟踪即可。

### 2. 枚举转换条件

每个状态 + 输入 → 下一个状态。写出来就是一目了然的决策表：

```text
UNKNOWN  + banner 含 "login:"              → WAITING_USERNAME
UNKNOWN  + 探测含 __SH_STATUS_PROBE__       → NO_LOGIN_REQUIRED
UNKNOWN  + 探测含 Password: 且含 login:      → WAITING_USERNAME
UNKNOWN  + 探测含 Password: 不含 login:      → (二次探测)
UNKNOWN  + 探测含 incorrect                 → WAITING_USERNAME
UNKNOWN  + 其他                             → ERROR
(二次探测) + 输出含 __SH_STATUS_PROBE__      → NO_LOGIN_REQUIRED
(二次探测) + 输出含 incorrect / Password:    → WAITING_USERNAME
(二次探测) + 其他                            → ERROR
```

注意第 4 行和第 8 行：同一套判断逻辑走了两遍——第一次输出的“模糊”促使了第二次探测。状态机里体现为自循环：`(探测中) + (模糊) → (继续探测)`。

### 3. 设计对外接口

调用方不需要知道状态机内部怎么判断，只需要两个动作：

- `start(banner)` —— 喂入初始数据，拿到第一个动作指令
- `feed(output)` —— 喂入每次读取的输出，拿到下一动作指令

每个动作指令告诉调用方两件事：

- `done: boolean` —— 是否到达终态（不需再喂数据）
- `send?: string` —— 如果还没到终态，下一步要发什么命令

```typescript
interface StateMachineAction {
  send?: string;                    // 下一步要发送的命令（undefined = 已达终态）
  waitMs: number;                   // 发送后等待多久
  state: UserLoginStatus;           // 当前状态
  done: boolean;                    // 是否终态
}
```

调用方只需要一个简单循环：

```typescript
const sm = new UserLoginStateMachine();
let action = sm.start(banner);

while (!action.done) {
  channel.write(action.send!, 1);          // 写之前清缓冲区
  await wait(action.waitMs);
  action = sm.feed(channel.read(1));       // 喂入输出，拿下一指令
}

// 终态处理
if (action.state === NO_LOGIN_REQUIRED) { /* 已登录 */ }
else if (action.state === WAITING_USERNAME) { /* 走登录 */ }
else { /* 异常 */ }
```

### 4. 画状态图

在写代码之前，**用图理清思路**比直接写代码更高效。画图时会发现遗漏的路径：

- “二次探测还是模糊怎么办？” → 加了 `_probeCount` 上限
- “同时含 Password: 和 login: 走哪个？” → 明确了优先级排序
- “探测被吞后终端回了 login:，还需要二次探测吗？” → 发现可以短路跳过，节省一轮往返


## 三、 UserLoginStateMachine 实现要点

`UserLoginStateMachine` 在 `src/transport/user-login.ts` 文件中声明，负责终端状态探测，不处理实际的用户名/密码交互。

### 1. 职责边界

| 负责 | 不负责 |
|---|---|
| 判断终端处于什么状态 | 发送用户名和密码 |
| 发送探测命令并分析回显 | 验证登录凭据 |
| 决定“需要登录”还是“已登录” | 执行登录交互序列 |

登录交互由 `UserLoginHandler` 完成，状态机的终点 `WAITING_USERNAME` 即是 `UserLoginHandler` 的起点。

### 2. 关键设计决策

【**探测次数上限**】二次探测后仍模糊不清，不再无限重试，返回 `ERROR`。防止因串口噪声、设备异常等导致的死循环。

```typescript
if (this._probeCount > 2) {
  return this.#reply(UserLoginStatus.ERROR, "探测次数超限");
}
```

【**短路优化**】当一次探测输出同时含 `Password:` 和 `login:`，说明探测命令被当密码吃掉、验证失败、终端已回到登录提示。此时直接返回 `WAITING_USERNAME`，不需要再发一次探测。

【**不含 login: 的进一步区分**】如果含 `Password:` 但不含 `login:`，无法判断是“探测被吞但终端还在等密码”还是“探测被吞但密码正确已登录”。此时需要二次探测——再发一次 echo，从新一轮输出中判断。

### 3. 关键方法

#### 3.1 start()

该函数在 `src/transport/user-login.ts` 文件中声明：

```typescript
start(banner: string): StateMachineAction
```

【**函数作用**】用串口/SSH 连接后的初始输出（banner）初始化状态机，返回第一个动作指令。

【**参数含义**】

- `banner`：连接后读取到的终端初始输出

【**返回值**】

- banner 含 `login:` → 返回终态 `WAITING_USERNAME`（调用方直接走登录）
- 其他情况 → 返回探测动作（调用方发 echo 命令）

#### 3.2 feed()

该函数在 `src/transport/user-login.ts` 文件中声明：

```typescript
feed(output: string): StateMachineAction
```

【**函数作用**】将终端输出喂入状态机，状态机根据当前状态和输出内容决定下一步动作。

【**参数含义**】

- `output`：从通道读取到的终端输出文本

【**返回值**】

- 能明确判断 → 返回终态（`NO_LOGIN_REQUIRED` / `WAITING_USERNAME` / `ERROR`）
- 模糊不清 → 返回二次探测动作（调用方再发一次 echo）

## 四、 扩展指南

### 1. 添加新的登录样式

如果遇到新的设备登录提示符（如 `"Username:"` 而不是 `"login:"`），只需在 `BUILTIN_PROFILES` 中添加新的 profile 条目。状态机本身不需要修改。

### 2. 添加新的状态

如果设备支持更复杂的交互流程（如二次认证、验证码），步骤：

（1）在 `UserLoginStatus` 枚举中添加新状态

（2）在 `feed()` 中添加对应转换规则

（3）更新状态图注释

### 3. 非串口场景复用

`UserLoginStateMachine` 依赖 `UserLoginChannel` 接口（`write` / `read`），不绑定串口。SSH 通道同样可用：

```typescript
const sm = new UserLoginStateMachine(profile);
let action = sm.start(sshBanner);

while (!action.done) {
  await sshShell.write(action.send!);
  await wait(action.waitMs);
  action = sm.feed(await sshShell.read());
}
```

## 五、 总结

状态机不是银弹，但当逻辑出现“分阶段、重输入、可重试”特征时，它比嵌套 if-else 更清晰：

- **状态**：枚举所有可能的情况
- **转换**：明确每个输入在每种状态下导致什么结果
- **接口**：调用方只管“喂输入、拿指令”循环
- **图先行**：画图理清思路，再写代码
