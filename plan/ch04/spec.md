# SSH 文件传输工具（SFTP）Spec

## 背景

`embedded-mcp-toolkit` 当前 SSH 类 MCP 工具只支持交互式 shell（`ssh_shell_open/close/write/read/exec/login`）和远端编译（`ssh_build`），**不支持文件传输**。实际使用中，经常需要在开发机与板卡之间搬运成果物（固件、镜像、工具链产物等），部分文件较大，可达上百 MB。

底层 SSH 传输层用的是 `ssh2` 库，它**原生支持 SFTP 子系统**（`client.sftp()` → `fastGet/fastPut` 流式传输），无需引入新依赖。但当前 `SSHShell` 类的 ssh2 `Client` 是私有字段（`#client`），且只暴露了 shell 通道，未暴露 SFTP 能力。

## 目标

- 在**已建立的 SSH 会话**上提供文件上传/下载能力，复用同一条 TCP+SSH 连接，避免重复握手。
- 支持大文件（上百 MB）稳定、流式传输，不在内存中缓冲整个文件。
- 不引入新依赖（`ssh2` 已内置 SFTP）。

## 功能需求

- **F1：上传工具** — 在指定 `session_id` 的 SSH 连接上发起 SFTP 子系统，将本地文件传输到远端板卡指定路径。
- **F2：下载工具** — 在指定 `session_id` 的 SSH 连接上发起 SFTP 子系统，将远端板卡文件拉取到本地指定路径。
- **F3：会话复用** — 复用 `ssh_shell_open`/`ssh_shell_login` 建立的会话，通过 `session_id` 定位已有连接，不单独建立新连接。
- **F4：传输摘要** — 传输完成后返回摘要信息：源路径、目标路径、传输字节数、耗时、平均速率、成功/失败状态。
- **F5：SSHShell SFTP 能力扩展** — 扩展 `SSHShell` 类暴露 SFTP 子系统访问能力（按需懒加载 SFTP 会话），供文件传输工具调用，不影响现有 shell 功能。

## 非功能需求

- **N1：不引入新依赖** — `ssh2` 已内置 SFTP 子系统支持，本次仅使用项目已有依赖。
- **N2：流式传输，不撑内存** — 大文件（上百 MB）通过 `fastGet`/`fastPut` 流式传输，不在内存中缓冲整个文件内容。
- **N3：不影响现有 shell 功能** — 扩展 `SSHShell` 时，SFTP 子系统与已有 shell 通道共存于同一连接；现有 `open/write/read/close` 行为不变。
- **N4：懒加载 SFTP** — SFTP 会话按需创建，不在 `open()` 时就建立；`close()` 时一并释放 SFTP 资源。
- **N5：错误以文本返回** — 与现有工具一致，传输失败、文件不存在、会话不存在等情况以文本内容返回，不抛异常中断。
- **N6：代码风格一致** — 与 `src/transports/ssh.ts`、`src/mcp/tools/ssh/shell.ts` 现有风格一致（JSDoc 注释、`.js` 后缀 import、命名约定）。
- **N7：保持 session 元数据一致性** — 文件传输不创建新会话，复用现有 `session_id`；不改动 `SessionRegistry`。

## 不做的事

- **不做中间进度推送** — MCP 工具调用是同步阻塞模型，传输完成前客户端看不到中间输出；只返回最终摘要（受模型限制，N 章节已说明）。
- **不做断点续传** — 本次只做整文件传输，不支持从上次中断位置续传。
- **不做目录递归传输** — 只传输单个文件，不递归上传/下载目录（`fastGet`/`fastPut` 针对单文件）。
- **不做权限/属主修改** — 传输后不自动 `chmod`/`chown`，如需修改由用户用 shell 工具另做。
- **不新建独立 SFTP 会话类型** — 不引入 `ssh_sftp_open` 这类单独建连的工具，全部复用现有会话。
- **不改动设备配置格式** — SFTP 复用现有 `getSSHConfig` 产出的连接，不新增配置项。

## 验收标准

- **AC1（F1）**：调用上传工具，传入一个已打开的 `session_id`、本地文件路径、远端目标路径，工具执行后远端目标路径出现该文件，且字节数与本地源文件一致。
- **AC2（F2）**：调用下载工具，传入一个已打开的 `session_id`、远端文件路径、本地目标路径，工具执行后本地目标路径出现该文件，且字节数与远端源文件一致。
- **AC3（F3）**：传入不存在的 `session_id` 时，工具返回 "Session xxx not found." 文本，不抛异常。
- **AC4（F3）**：上传/下载在 `ssh_shell_open`/`ssh_shell_login` 返回的会话上执行，无需单独建立新 SSH 连接（验证：传输前后 `ssh_shell_exec` 仍可在同一 session 正常工作）。
- **AC5（F4）**：传输成功后，工具返回文本摘要，至少包含：传输方向、源路径、目标路径、字节数、耗时（毫秒）、平均速率。
- **AC6（F4）**：源文件不存在时，工具返回清晰的错误文本（包含远端/本地路径与失败原因），不抛异常、不产生半成品文件。
- **AC7（F5）**：`SSHShell` 新增 SFTP 能力后，现有 `ssh_shell_open/close/write/read/exec/login/build` 七个工具行为不变（回归验证，编译通过 + 一次 shell 往返正常）。
- **AC8（N2）**：上传/下载一个约 50–100MB 文件时，MCP server 进程内存占用无显著峰值上涨（流式传输，非全量读入内存）。
- **AC9（N3/N4）**：同一会话先做 shell 操作，再做文件传输，再做 shell 操作，三次操作均成功——证明 SFTP 与 shell 通道在同连接上共存。
- **AC10（N4）**：调用 `ssh_shell_close` 关闭会话后，该会话的 SFTP 资源随之释放，不再占用连接。
