## 一、 概述

`ssh_build` 是 embedded-mcp-toolkit 提供的远程编译工具，用于在嵌入式板卡上通过 SSH 会话执行长时间编译命令（如 `make`、`cmake --build`、shell 脚本等），并返回结构化编译结果。

与 `ssh_shell_exec`（发送命令后固定延迟读取）不同，`ssh_build` 会持续轮询直到编译完成，自动分类采集错误、警告和常规信息，适合耗时可能超过数分钟的编译任务。

## 二、 架构与方案设计

### 1. Client-Server 架构

embedded-mcp-toolkit 作为 MCP Server，通过 stdio 与 MCP Client（opencode / Claude Code）通信。Client 发起 `tools/call`，Server 执行并返回结果：

```text
┌──────────────────────────────────┐      ┌────────────────────────────────────┐
│  MCP Client (opencode/Claude)    │      │  MCP Server (embedded-mcp-toolkit) │
│                                  │      │                                    │
│  LLM 推理 → tools/call ───────────│─stdio│──→ ssh_shell_open                  │
│                                  │◄─────│── 返回 session_id                  │
│                                  │      │                                    │
│  LLM 推理 → tools/call ───────────│──────│──→ ssh_build(session_id, command, ...) │
│                                  │      │   ├─ 发送编译命令                   │
│                                  │      │   ├─ 轮询检测完成标记               │
│                                  │      │   ├─ 分类错误/警告/信息             │
│                                  │      │   └─ 组装结构化结果                 │
│                                  │◄─────│── 返回 BUILD SUCCESS/FAILED         │
└──────────────────────────────────┘      └────────────────────────────────────┘
```

编译工具的实现面临一个关键设计决策：**轮询和完成判断逻辑应该放在哪一端？**

### 2. 两种处理模式

#### 2.1 Client 端轮询模式（方案 A）

让 opencode 或 Claude Code 反复调用 `ssh_shell_write` → `ssh_shell_read`，由 LLM 在每次响应后自行判断编译是否完成。

```text
Client 端轮询模式（一次编译 = N 次工具调用）：
  tools/call ssh_shell_write("make -j8")
      → 响应: ok
  tools/call ssh_shell_read()          ← LLM 判断: 还没结束
      → 响应: "compiling foo.c..."
  tools/call ssh_shell_read()          ← LLM 判断: 还没结束
      → 响应: "compiling bar.c..."
  ...（重复 N 轮，每轮 LLM 消耗 token 做判断）
  tools/call ssh_shell_read()          ← LLM 判断: 看到提示符了，结束
      → 响应: "compilation done #"
```

#### 2.2 Server 端封装模式（方案 B）

由 MCP Server 内部轮询完成标记，一次 `tools/call` 返回完整结果。

```text
Server 端封装模式（一次编译 = 1 次工具调用）：
  tools/call ssh_build("make -j8")
      → Server 内部: 发送命令 → 轮询检测标记 → 分类 → 组装
      → 响应: "BUILD SUCCESS (exit 0) | 0 errors, 2 warnings | <完整日志>"
```

#### 2.3 对比总结

| 维度 | 方案 A：Client 端轮询 | 方案 B：Server 端封装 |
|------|----------------------|---------------------|
| 工具调用次数 | N 次（N ≈ 编译耗时 / 轮询间隔） | **1 次** |
| LLM 参与度 | 每轮都需推理判断是否结束 | **零干预**，直接拿到结果 |
| Token 消耗 | 每次轮询消耗 token，N 轮可观的累积成本 | **仅 1 次响应** |
| 误判风险 | LLM 可能把输出中的 `error` 当编译失败提前终止，或把 `BUILD` 字样当结束 | **无**，代码确定性判断 |
| 结束判断方式 | 靠 LLM 猜（提示符/输出停滞） | **完成标记**精确匹配 |
| 退出码获取 | LLM 需额外执行 `echo $?` | **内嵌在标记中**，一次获取 |
| 日志完整性 | LLM 需自行拼凑 N 轮读取结果 | **Server 自动拼接** |
| 错误/警告分类 | LLM 需自行分析 | **Server 自动分类** |

### 3. 选择 Server 端处理？

综合考虑上述对比，本工具选择 Server 端封装模式，具体原因如下：

#### 3.1 MCP 协议是请求-响应模型

MCP `tools/call` 是单次请求-单次响应的模式，不支持流式长连接。Client 端轮询意味着每次 `ssh_shell_read` 都是一次独立的 `tools/call`，每次都需要完整的请求-响应往返。

#### 3.2 Token 消耗与上下文膨胀

这是 Client 端轮询最致命的问题。下面从 token 消耗和上下文窗口两个维度定量分析。

##### 3.2.1 基础概念

在深入量化分析之前，先澄清两个核心概念：Token 和上下文窗口。

【**1. 什么是 Token？**】

Token 是 LLM 处理文本的最小单位，可以是单词、汉字的一部分，或一个标点符号。API 按 token 计费，通常分为三类：

- **输入 token**：你发送给模型的全部内容（系统提示、工具定义、历史对话、当前消息）。
- **输出 token**：模型生成并返回给你的内容。
- **缓存 token / 缓存命中**：部分平台（如 Claude）会对重复出现的输入做 Prompt Caching。如果某段文本在极短时间内被再次发送，平台可能只收取较低的缓存费用，甚至不计入新增输入 token，称为“缓存命中”。

> **示例（3 轮对话）**：假设上下文窗口上限为 8K tokens，平台支持 Prompt Caching。
>
> 每轮 API 请求的结构是：`系统提示 + 工具定义 + 历史消息（user/assistant 对）+ 当前用户消息`。其中系统提示和工具定义每轮都**单独发送**，不属于"历史"。
>
> ```text
> 第 1 轮：初始请求
> ┌─────────────────────────────────────────────────────────┐
> │ 输入                                                    │
> │   [系统提示]      500 tokens   ──→ 首次发送，正常计费    │
> │   [工具定义]    3,000 tokens   ──→ 首次发送，正常计费    │
> │   [用户消息1]      10 tokens   ──→ 新输入，正常计费      │
> │   ─────────────────────────────────────────             │
> │   本轮输入合计: 3,510 tokens                            │
> └─────────────────────────────────────────────────────────┘
>                          ↓ 模型生成
> ┌─────────────────────────────────────────────────────────┐
> │ 输出                                                    │
> │   [模型回复1]     200 tokens   ──→ 输出 token            │
> └─────────────────────────────────────────────────────────┘
> 窗口占用: 3,510 + 200 = 3,710 tokens  (窗口剩余 ~54%)
> 
> ─────────────────────────────────────────────────────────
> 
> 第 2 轮：继续对话
> ┌─────────────────────────────────────────────────────────┐
> │ 输入                                                    │
> │   [系统提示]      500 tokens   ──→ 与上轮相同 → 缓存命中 │
> │   [工具定义]    3,000 tokens   ──→ 与上轮相同 → 缓存命中 │
> │   ── 历史（第 1 轮的 user/assistant 对）──               │
> │   [用户消息1]      10 tokens   ──→ 历史重传，正常计费    │
> │   [模型回复1]     200 tokens   ──→ 历史重传，正常计费    │
> │   ── 当前消息 ──                                         │
> │   [用户消息2]      12 tokens   ──→ 新输入，正常计费      │
> │   ─────────────────────────────────────────             │
> │   本轮输入合计: 3,722 tokens (其中 3,500 为缓存计费)    │
> └─────────────────────────────────────────────────────────┘
>                          ↓ 模型生成
> ┌─────────────────────────────────────────────────────────┐
> │ 输出                                                    │
> │   [模型回复2]     250 tokens   ──→ 输出 token            │
> └─────────────────────────────────────────────────────────┘
> 窗口占用: 3,722 + 250 = 3,972 tokens  (窗口剩余 ~50%)
> 
> ─────────────────────────────────────────────────────────
> 
> 第 3 轮：再次追问
> ┌─────────────────────────────────────────────────────────┐
> │ 输入                                                    │
> │   [系统提示]      500 tokens   ──→ 缓存命中              │
> │   [工具定义]    3,000 tokens   ──→ 缓存命中              │
> │   ── 历史（前两轮的 user/assistant 对）──                │
> │   [用户消息1]      10 tokens   ──→ 历史重传，正常计费    │
> │   [模型回复1]     200 tokens   ──→ 历史重传，正常计费    │
> │   [用户消息2]      12 tokens   ──→ 历史重传，正常计费    │
> │   [模型回复2]     250 tokens   ──→ 历史重传，正常计费    │
> │   ── 当前消息 ──                                         │
> │   [用户消息3]      10 tokens   ──→ 新输入，正常计费      │
> │   ─────────────────────────────────────────             │
> │   本轮输入合计: 3,982 tokens (其中 3,500 为缓存计费)    │
> └─────────────────────────────────────────────────────────┘
>                          ↓ 模型生成
> ┌─────────────────────────────────────────────────────────┐
> │ 输出                                                    │
> │   [模型回复3]     150 tokens   ──→ 输出 token            │
> └─────────────────────────────────────────────────────────┘
> 窗口占用: 3,982 + 150 = 4,132 tokens  (窗口剩余 ~48%)
> (若继续追问几轮将触及 8K 上限，最早的消息会被丢弃)
> ```
>
> 从 3 轮对话可以看出：
>
> - **输入 token**：每轮发送的全部内容，且必须携带完整历史（第 2 轮起历史占了大头）。
> - **输出 token**：模型每轮生成的回复，独立计费。
> - **缓存命中**：系统提示和工具定义重复发送时，平台按缓存 token 低价计费；但**历史对话不会被缓存**，每轮都要全额重传。
>
> 为什么历史对话不会被缓存？原因有两点：
>
> 1. **缓存机制基于前缀精确匹配**：Prompt Caching 的原理是复用请求开头与最近某次请求完全一致的部分。系统提示和工具定义是静态前缀，因此能命中缓存；但历史对话是每轮都在变化的后缀，无法匹配任何已缓存版本。
> 2. **历史是动态累积的**：第 2 轮的历史包含第 1 轮全部内容，第 3 轮又包含前两轮全部内容。每一轮的历史都是从未出现过的全新文本序列，平台没有可复用的缓存副本。
>
> 简言之：**静态内容可缓存，动态累积的历史不可缓存**。这也是 Client 端轮询模式下历史重传成为 token 消耗大头的原因。
>
> - **上下文窗口**：它像一条“记忆滑轨”，累计历史 + 当前输入 + 当前输出不能超过上限，否则旧内容会被挤出，模型“遗忘”早期对话。

【**2. 什么是上下文窗口？**】

上下文窗口（Context Window）指模型在一次请求中能够“看到”的最大 token 数量，**包括输入和输出**。它决定了模型能记住多少历史信息。

上下文窗口与 token 的关系可以用“记忆力”来类比：token 是记忆的“原子单位”，上下文窗口是记忆的“总容量”。如果你的对话历史累计超过了窗口上限，最早的消息就会被丢弃或压缩，模型因此“遗忘”了前面的内容。

> **示例（窗口上限 12K tokens）**：
>
> ```text
> 上下文窗口示意（每个 █ 代表 1K tokens，窗口共 12 格）：
> 
> 第 1 步：发送 8K 文档，要求总结
> ┌─────────────────────────────────────────────────────┐
> │ 窗口总容量: 12K                                      │
> │ [文档A ████████░░░░]  8K 输入                         │
> │                      ↑ 模型生成                       │
> │ [总结A ███░░░░░░░░░]  3K 输出                         │
> │ ─────────────────────────────────────────             │
> │ 已用 11K / 12K  (剩余 1K，未超限)                     │
> └─────────────────────────────────────────────────────┘
> 
> 第 2 步：追加 5K 新文档，要求对比分析
> ┌─────────────────────────────────────────────────────┐
> │ 理论上需要容纳的全部内容：                             │
> │ [文档A ████████░░░░]  8K  ← 第1轮输入                │
> │ [总结A ███░░░░░░░░░]  3K  ← 第1轮输出                │
> │ [文档B █████░░░░░░░]  5K  ← 第2轮新输入              │
> │ ─────────────────────────────────────────             │
> │ 实际需求: 16K  >  窗口上限: 12K   ⚠️ 溢出 4K！        │
> └─────────────────────────────────────────────────────┘
> 
> 平台实际处理（截断最早内容）：
> ┌─────────────────────────────────────────────────────┐
> │ [文档A ██░░░░░░░░░░]  2K  ← 文档A被截断，只剩尾部    │
> │ [总结A ███░░░░░░░░░]  3K  ← 保留                     │
> │ [文档B █████░░░░░░░]  5K  ← 新输入必须保留           │
> │ ─────────────────────────────────────────             │
> │ 已用 10K / 12K  (预留 2K 给模型输出)                 │
> │                                                      │
> │ ❌ 后果：模型看不到文档A的开头，若用户问"A的第一段     │
> │    讲了什么"，模型无法回答，因为那段已经被挤出窗口了。 │
> └─────────────────────────────────────────────────────┘
> ```
>
> 这个示例说明：上下文窗口是**硬性上限**，不是“可以协商的软限制”。一旦内容总量超过窗口容量，最早的内容会被静默截断，模型对此毫无感知，只会基于剩余的内容作答。

理解了这两个概念后，就能明白为什么 Client 端轮询会导致成本飙升和上下文丢失。

##### 3.2.2 Token 消耗

LLM API 按请求的输入+输出 token 计费。关键在于：**每次 `tools/call` 请求都需携带完整对话历史**，因此单轮请求 token 数随轮次递增，总消耗呈 O(N²) 增长——而非直觉上的 O(N)。

假设一次典型编译持续 5 分钟，2 秒轮询间隔（N=150 轮），每轮新增约 1,000 tokens 对话历史（工具调用+日志增量+LLM 推理+消息框架）：

```text
方案 A（Client 端轮询，150 次 read）：
  每轮请求都需重新发送 系统+工具定义+全部历史+当前消息：
  第 1 轮: 输入 ≈  4,000 tokens (系统+工具+当前消息，无历史)
  第 2 轮: 输入 ≈  5,000 tokens (系统+工具+1轮历史+当前消息)
  第 3 轮: 输入 ≈  6,000 tokens (系统+工具+2轮历史+当前消息)
  ...
  第 i 轮: 输入 ≈  4,000 + 1,000×(i-1)
  第 150 轮: 输入 ≈  153,000 tokens
  每轮 LLM 输出: ~300 tokens

  总消耗 ≈ Σ[4,000 + 1,000×(i-1)] + 300×150  (i=1..150)
         = 4,000×150 + 1,000×(0+1+...+149) + 45,000
         = 600,000 + 11,175,000 + 45,000
         ≈ 11,820,000 tokens

方案 B（ssh_build，1 次调用）：
  请求: ~4,000 tokens（系统提示 + 工具定义 + 参数）
  响应: ~8,000 tokens（结构化结果）
  单次合计: ~12,000 tokens

Token 消耗比: 11,820,000 / 12,000 ≈ 985 倍
```

【**注意**】

以上为保守估算，实际差距可能更大：

- 每轮对话历史增量按 1,000 tokens 计，编译输出密集时可能更高
- 编译日志实际通常远大于 2000 行（Linux 内核编译可达数万行），增量日志累计更大
- 系统+工具定义实际可能超过 3,000 tokens（50+ 工具定义本身较大），每轮请求都需重传

##### 3.2.3 上下文窗口膨胀

LLM 的上下文窗口限制了单次请求能处理的最大 token 数（如 Claude 的 200K）。Client 端轮询模式下，每轮请求必须包含完整对话历史，上下文随轮次线性增长：

```text
方案 A（Client 端轮询）在第 N 轮时的上下文构成：
  ├─ 系统提示 + 工具定义:  ~3,000 tokens   ← 出现 1 次，不随轮次累积
  ├─ 对话历史:             ~1,000 × N       ← N 轮交互记录线性累积
  │   （每轮含工具调用 + 日志增量 + LLM 推理 + 消息框架）

  以 N=150 为例:
  系统 + 工具定义:    ~3,000 tokens
  对话历史:           ~150,000 tokens   ← 大头
  ──────────────────────────────
  第 150 轮上下文:    ≈ 153,000 tokens  → 接近但尚未超限

  以 N=200 为例:
  第 200 轮上下文:    ≈ 203,000 tokens  → 超过 200K 上限，请求失败

方案 B（ssh_build）的上下文构成（1 轮）：
  ├─ 系统提示 + 工具定义:  ~3,000 tokens
  ├─ 结构化结果:           ~8,000 tokens  ← 已分类压缩
  └─ 总上下文:              ~11,000 tokens
```

**【注意】**

- **上下文窗口**指 LLM 单次推理能处理的最大 token 数（如 Claude 200K）。它约束的是"这一轮 LLM 能看多少字"——系统提示和工具定义在单轮上下文中只出现 1 次，不随轮次叠加
- **API 计费**按每次 HTTP 请求的输入+输出 token 总量计费。工具定义随每次请求发送，因此每轮都计入费用
- 两者区别：上下文窗口是"单轮容量上限"，API 计费是"累计传输量"。Client 端轮询模式下，工具定义虽不叠加窗口容量，但因每轮重传，计费仍被多次计算
- 上下文膨胀的真正驱动力是**对话历史**（每轮交互约 1,000 tokens，N 轮即约 1,000×N tokens）
- 编译时间越长（N 越大），越早触及窗口上限，且一旦超限请求直接失败
- 如果误用 `clear=0`（每次返回全部累积输出而非增量），每轮的日志内容本身也会随轮次增长，对话历史呈二次膨胀：第 i 轮日志长度 ≈ i/150 × 总日志量，N 轮累计 ≈ `日志总量 × (N+1)/2`，远更早触及上限

##### 3.2.4 对比总结

| 维度 | 方案 A：Client 端轮询 | 方案 B：ssh_build |
|------|----------------------|-------------------|
| 工具调用次数 | 150 次 | 1 次 |
| Token 总消耗（5 分钟编译） | ~11,820,000（O(N²) 累积，历史重传） | ~12,000（O(1)） |
| 工具定义发送次数 | 150 次（每次请求重传） | 1 次 |
| 工具定义在上下文中累积 | 1 次（属于系统提示，不累积） | 1 次 |
| 是否会撑爆上下文 | **是**（约 200 轮后超 200K） | 否 |
| 日志内容在上下文中出现 | N 次（分散在对话历史中） | 1 次 |

#### 3.3 完成拉长轮询间隔能解决吗？

如果把 Client 端轮询间隔从 2 秒拉长到 5 分钟，确实能减少调用次数从而缓解 token 和上下文压力。但这引入新问题：

- **不知道什么时候开始查**：LLM 无法预知编译耗时。如果增量编译只需 3 秒，得干等 5 分钟才第一次读，无谓浪费；如果设短了又回到频繁调用
- **仍需完成标记**：5 分钟读一次，如何判断日志中"编译完了"还是"还在跑只是没输出"？最终还是需要一个标记。标记在 Server 端已构造好，却要传给 Client 让 LLM 做正则匹配——纯属多此一举

拉长间隔只是把"低效方案"变得"不那么低效"，永远不如 Server 端直接检测。

#### 3.4 误判风险

- LLM 可能把编译输出中的 `"error: file not found"` 理解为"编译出错了"，提前终止轮询
- LLM 可能看到 `"BUILD"` 字样或提示符就误认为结束
- 无完成标记时，LLM 只能靠"提示符出现了"或"输出不变化了"来猜测——这些方式在第三章中已证明不可靠

#### 3.5 LLM 不适合确定性循环检测的深层原因

LLM 的核心能力是**理解与生成**，而非**循环控制**。判断"日志里有没有出现 `___MCP_BUILD_DONE___`"是一个确定性正则匹配任务，用代码完成是**毫秒级、零 token** 的；让 LLM 来做，完全是用错了工具。

#### 3.6 语义原子性

`ssh_build` 将"发送命令 + 轮询等待 + 检测完成 + 分类输出"封装为一个原子操作：

```text
Server 端: ssh_build = 发送命令 + 轮询等待 + 检测完成 + 分类输出
Client 端: 一次工具调用，拿到结构化结果，AI 直接分析
```

如果拆分给 Client 轮询，则"编译"这个原子操作被切割为 `ssh_shell_write` + N 个 `ssh_shell_read`，语义碎片化，AI 需要自行拼凑完整日志、判断编译成功与否、手动分类错误和警告。

#### 3.7 外部验证：Claude Code Bash 工具的同构设计

Claude Code 的 `Bash` 工具同样需要在远端执行命令并判断完成，其方案同样是**在命令末尾注入标记**：

```
<命令>; echo __claude_code_bash_done__:$?
```

而非让 LLM 去匹配提示符或轮询输出。这从侧面印证：**Server 端完成标记方案是当前主流 AI 工具的共识选择**。

## 三、 完成判断方案对比

确定了在 Server 端处理后，下一个问题是：**Server 内部如何判断编译已经完成？** 

### 1. 候选方案

| 方案 | 可行性 | 问题 |
|------|--------|------|
| 检测 shell 提示符（`#`/`$`） | ❌ | 提示符格式多变（`#`/`$`/`>`/自定义），且可能出现在编译输出中（如 make 回显 shell 变量），不可靠 |
| 检测输出停滞（安静期） | ❌ | 编译过程（尤其是链接阶段）可能有数分钟无输出，无法区分"还在编译"和"已结束"；需设置超长且不可预测的安静期阈值 |
| 检测产物是否生成 | ❌ | `ssh_build` 是通用工具，不知道产物名称和路径；文件可能在编译中途就已创建（如 Makefile 规则提前 `touch` 产物），产生假阳性 |
| 检测编译脚本结束消息 | ❌ | 不同构建系统打印不同文本（`make` 报 `make: ***`，CMake 报 `Build files have been written`，自定义脚本各写各的），无统一模式；部分脚本静默退出，不打印任何结束消息；无法可靠提取退出码 |
| **完成标记** | ✅ | 唯一字符串 `___MCP_BUILD_DONE___` 不外显于编译日志，零误判；内嵌 `$?` 一次获取退出码；不依赖 shell 状态、构建系统类型或编译时长 |

### 2. 完成标记方案原理

将编译命令包装为：

```shell
// 无工作目录时：
<用户命令> 2>&1; echo "___MCP_BUILD_DONE___:$?"

// 有工作目录
cd <cwd> || { echo "___MCP_BUILD_DONE___:1"; exit 1; }; <用户命令> 2>&1; echo "___MCP_BUILD_DONE___:$?"
```

- `;` 是 Shell 顺序执行操作符，前后两条命令严格按先后顺序执行：`echo` 必须等用户命令执行完毕后才会执行，因此标记出现在输出中即意味着编译已结束
- `2>&1` 将标准错误合并到标准输出，确保所有日志在同一流中
- `$?` 是 shell 中上一条命令的退出码（0 = 成功，非 0 = 失败）
- 编译结束后 `echo` 打印唯一标记 + 退出码
- 有 `cwd` 时先执行 `cd`，若 `cd` 失败则立即输出标记（退出码 1）并 `exit 1`，不执行后续编译
- cd 失败时 `echo marker` 的原因是：若只用 `cd ... || exit 1`，shell 直接终止，标记不会输出，处理器将无数据可匹配而一直等到超时（默认 10 分钟）；加上 `echo marker` 后 cd 失败也能立即返回退出码，实现快速报错

Server 发送命令后进入轮询循环（默认每 2 秒一次），从 SSH 会话缓冲区增量读取输出，用正则 `/___MCP_BUILD_DONE___:(\d+)/` 匹配标记。匹配到即为编译结束，标记之前的所有内容即为完整编译日志。

### 3. 完成标记的优势

- **确定性**：标记严格等于命令结束的那一刻，无歧义
- **通用性**：适用于任意 shell 命令，无需预先知道构建系统类型、脚本格式或产物名称
- **退出码内嵌**：`$?` 随标记一并返回，一次正则匹配同时得到"是否结束"和"成功/失败"
- **零误判**：标记字符串足够唯一，不会在常规编译输出中出现

## 四、 核心机制

### 1. 完成标记检测

命令构造逻辑位于 `buildRemoteCommand()`：

- 将用户命令的标准输出与标准错误合并（`2>&1`）
- 命令执行完毕后，`echo` 输出唯一标记字符串 `___MCP_BUILD_DONE___` 及其退出码 `$?`
- 如果指定了工作目录，先 `cd` 到目标目录；`cd` 失败则 `echo` marker 退出码 1 并 `exit 1`，不执行后续编译

标记构造完成后，首先剥离 PTY 回显（见 2 节），然后进入轮询循环检测标记。用正则 `/___MCP_BUILD_DONE___:(\d+)/` 匹配，匹配到后提取退出码，标记之前的内容即为完整日志。

### 2. PTY 回显剥离

#### 2.1 问题现象

测试中发现 `ssh_build` 发送编译命令后仅 2 秒就返回 `BUILD FAILED (exit 1)`，仅收集到 39 字节输出，而远端构建脚本（`./build.sh`）实际仍在正常执行。服务端日志片段如下：

```text
[17:59:19] [INFO] [ssh_build] command=./build.sh alpha -a -c
[17:59:19] [OSC]sumu@virtual-machine:~$ cd ~/workspace/Alpha/kernel || { echo "___MCP_BUILD_DONE___:1"; exit 1; }; ./build.sh alpha -a -c 2>&1; echo "___MCP_BUILD_DONE___:$?"
[17:59:19] [INFO] 检查主机编译器...
[17:59:19] [SUCCESS] ✓ gcc-9 已找到
[17:59:19] [INFO] 开始清理构建环境...
[17:59:20]   CLEAN   .
[17:59:21] [INFO] [ssh_build] completed exitCode=1 outputLength=39
```

构建实际在执行（gcc-9 检查、clean 操作都在正常打印），但处理器在第一次轮询时就误判构建完成并提前返回。

#### 2.2 根本原因

PTY 会将用户输入的命令原样回显。构造的远端命令中 cd 失败分支 `echo "___MCP_BUILD_DONE___:1"` 包含的字面量 `:1` 被正则 `___MCP_BUILD_DONE___:(\d+)` 匹配：

```text
远端命令（由 buildRemoteCommand 构造）：
  cd <cwd> || { echo "___MCP_BUILD_DONE___:1"; exit 1; }; <command> 2>&1; echo "___MCP_BUILD_DONE___:$?"

PTY 回显（发送命令后收到的第一行数据，以 \n 结尾）：
  cd ~/workspace/... || { echo "___MCP_BUILD_DONE___:1"; exit 1; }; ... ./build.sh ... 2>&1; echo "___MCP_BUILD_DONE___:$?"\n
                                                                          ^^^^ 正则匹配 :1 → 误判构建完成
```

虽然 `$?` 在回显中是字面量、不匹配 `\d+`，但 cd 失败分支的 `:1` 本身是数字，`match()` 在回显行中立即命中，退出码被误读为 1，标记之前的所有构建输出被截断丢弃。

#### 2.3 方案演变

##### 2.3.1 方案 A：改变命令构造（已否决）

将 `:1` 改为 `:$((1))`。shell 算术展开 `$((1))` 在回显中保持字面量形式（不匹配 `\d+`），仅在执行时展开为数字 `1`。

```shell
cd <cwd> || { echo "___MCP_BUILD_DONE___:$((1))"; exit 1; }
```

- 优点：改动极小，一行解决
- 缺点：是补丁而非根治，若有任何需要 echo 含数字 marker 的 shell 代码片段，同样会触发误匹配；回避了 PTY 回显这个根本问题，而非正确面对它

##### 2.3.2 方案 B：回显剥离（采纳）

在检测完成标记之前剥离 PTY 回显行。实现方式经过三次迭代：

- **第一版（混合循环）**：在主轮询循环中用 `echoStripped` 标志位控制，`echoBuffer` 缓冲跨 chunk 回显，正则匹配 + `replace` 剥离，`continue` 跳过未完成时的后续逻辑。约 30 行，包含标志位、正则、分支、`continue`。
- **第二版（独立阶段 + 正则）**：将回显剥离与标记检测拆为两个独立循环，但回显行内仍用正则匹配命令原文再 `replace`。
- **第三版（独立阶段 + 直接截断，最终方案）**：回显剥离用一个独立的 `while` 循环，缓存到 `echoBuffer` 直到收到 `\n`，直接用 `substring(nlIdx + 1)` 截断丢弃整行——回显即命令原文本身，无需正则匹配。

#### 2.4 最终实现

```typescript
// 步骤 3：发送命令
// 用 clear=0（追加模式）：overflow=true 且 drain() 不重置，缓冲满时从头部覆盖，确保末尾标记不丢
shell.drain();
shell.write(fullCommand, 0);

// 步骤 4：剥离 PTY 回显
// 回显是发送命令后收到的第一行数据（以 \n 结尾），独立循环等待 \n 后直接截断
let allOutput: string = "";
let echoBuffer: string = "";
let echoRetries = 10;
while (echoRetries > 0) {
  echoRetries--;
  await new Promise(r => setTimeout(r, 200));
  echoBuffer += shell.drain();
  const nlIdx = echoBuffer.indexOf("\n");
  if (nlIdx !== -1) {
    allOutput = echoBuffer.substring(nlIdx + 1);
    break;
  }
}

// 步骤 5：轮询检测完成标记
const deadline = Date.now() + maxWait;
let exitCode = null;
const markerRegex = new RegExp("___MCP_BUILD_DONE___:(\\d+)");

while (Date.now() < deadline) {
  await new Promise(r => setTimeout(r, pollInterval));
  allOutput += shell.drain();
  const match = allOutput.match(markerRegex);
  if (match) {
    exitCode = parseInt(match[1], 10);
    allOutput = allOutput.substring(0, allOutput.search(markerRegex)).trimEnd();
    break;
  }
}
```

核心设计：

- `\n` 之前的所有内容 = PTY 回显（包含命令原文），整行丢弃
- `\n` 之后的内容 = 真实构建输出（包含构建日志以及 cd 失败时 `echo "___MCP_BUILD_DONE___:1"` 执行后的 marker），保留到 `allOutput`
- `echoRetries` 防止死循环：10 次 × 200ms = 2 秒超时，正常回显在第一次 200ms 内即可到达
- `write(cmd, 0)` 配合 `drain()` 不重置 overflow：构建期间缓冲区满时从头部覆盖旧数据，保留最新输出，确保末尾的完成标记不被丢弃
- 无需正则剥离：回显行就是命令本身，直接 `substring(nlIdx + 1)` 截断即可，不依赖正则匹配

### 3. 输出分类

标记检测成功后，编译日志按行分类为三类：

| 类别 | 匹配规则（正则，大小写不敏感） |
|------|----------|
| error | `\berror\b[:\s]`、`\bfatal\b[:\s]`、`undefined reference`、`No rule to make`、`make[...]: ***`、`cannot find`、`collect2: error`、`ld returned`、`\bfailed\b[:\s]`、`no such file or directory` |
| warning | `\bwarning\b[:\s]`、`\bwarn\b[:\s]`、`\bdeprecated\b`、`-W[a-z]+` |
| info | 未命中以上规则的其余行 |

分类逻辑优先匹配 error（更严重），其次 warning，其余归入 info。参数 `classify` 设为 `false` 时跳过分类，直接返回原始日志。

### 4. 结果组装

分类完成后按以下格式输出结构化结果：

```text

=== ERRORS (0) ===
(none)

=== WARNINGS (2) ===

[W2] src/baz.c:8:10: warning: unused variable 'x'

```

## 七、 超时注意事项

`ssh_build` 面临**两层超时**：

（1）**Server 端 `maxWait`**：默认 10 分钟，控制轮询等待上限。编译耗时超过此值则返回超时错误及已收集的部分输出

（2）**Client 端 MCP 超时**：取决于使用的 MCP Client
- opencode：默认 60 秒，需在 `opencode.json` 中配置 `mcp.<server>.timeout` 增大
- Claude Code：默认约 28 小时（`MCP_TOOL_TIMEOUT`），几乎不会触发

两层超时独立：即使 Server 端 `maxWait` 设为 60 分钟，若 Client 端 MCP 超时为 60 秒，编译也将在 60 秒后被 Client 掐断。详细原理见 `docs/MCP工具超时机制.md`。

## 八、 注意事项

- 每个 SSH 会话同一时间仅支持**一个**编译任务。需要并行编译时，应通过 `ssh_shell_open` 打开多个会话，各分配一个 `ssh_build`
- 编译命令通过 PTY 执行，输出包含 ANSI 转义序列（颜色码等），可能影响分类匹配
- 退出码仅反映编译命令的结果（0 或成功，`cwd` 切换失败时返回 1），不包括网络异常等通信层面的错误
