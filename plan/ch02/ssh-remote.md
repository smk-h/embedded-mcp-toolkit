## 一、 背景与问题

我在 Windows 上跑了一个 claudecode 或者 opencode，启动当前 MCP server，创建一个 SSH 会话连接到编译服务器，现在我想要使用 claude 或者 opencode 分析编译服务器的源代码。

> 已有上下文：[`vscode-sshremote.md`](vscode-sshremote.md) 分析了 VSCode Remote-SSH「无感远程开发」的实现原理——见「把无 UI 内核塞进远程、本地只留壳、用轻量 RPC 经 SSH 隧道交换消息」。本文讨论的是**另一个场景**：AI（claudecode / opencode）作为消费者，怎么拿到远端源代码来做分析。两者可以对照阅读。

当前 MCP server 已有 SSH 会话工具（`ssh_shell_login` / `ssh_shell_open` + `exec` / `read` / `write` + `close`），但这些工具**只能跑命令、回输出**，并没有「读文件内容到本地」这条直接路径。AI 要分析远端源代码，本质上只有三种思路：

```
思路 A: 把代码搬到 AI 能直接读的地方      (本地优先, 远端只作数据源)
思路 B: 把 AI 的"读/写"能力延伸到远端     (远端优先, 本地只作编排者)
思路 C: 让 AI 在远端就地分析              (远端跑 agent, 本地只发任务)
```

具体到落地，共 6 个主流方案，下面三节按这三类思路展开（第四、五、六节），最后给出方案选择建议。

## 二、 前置基础：MCP 的分层、stdio 与 SSH

在展开各方案前，先厘清三个贯穿全文的基础概念。后面每个方案都建立在这三层之上，这层搞清楚了，后续读到「MCP over SSH」「远端 stdio」时就不会卡壳。

### 1. MCP 的两层架构

MCP（Model Context Protocol）把协议分成两层，各管各的：

- **Data Layer（数据层）**：定义**消息长什么样、初始化怎么握手、有哪些能力、怎么列工具、怎么调工具**。这里的核心是 JSON-RPC 2.0，加上 lifecycle、tools / resources / prompts / notifications 这些语义。简单说就是「两边用什么格式的消息对话」。
- **Transport Layer（传输层）**：定义**消息怎么在两边之间传过去**。是走 stdio，还是走 Streamable HTTP，属于这一层。简单说就是「对话的载体是什么管道」。

```
┌─────────────────────────────────────────────┐
│  MCP client (agent: claude/opencode/zcode)  │
├─────────────────────────────────────────────┤
│  Data Layer    : JSON-RPC 消息 (tools/call) │
│  ─────────────────────────────────────────  │
│  Transport Layer: stdio | HTTP | ...        │
└─────────────────────────────────────────────┘
           ↑↓ Transport 管道
┌─────────────────────────────────────────────┐
│  MCP server (你启动的进程)                   │
├─────────────────────────────────────────────┤
│  Transport Layer: stdio | HTTP | ...        │
│  ─────────────────────────────────────────  │
│  Data Layer    : 处理 JSON-RPC, 执行工具     │
└─────────────────────────────────────────────┘
```

数据层和传输层**解耦**——同一套 JSON-RPC 消息（数据层不变），换根管子就能用（传输层可换）。这正是本文各方案能复用同一套消息语义、只换传输方式的理论基础：A1 / A2 / B2 的 server 跑在本地，C1 / C2 / C3 的 server 跑在远端，**对话内容一模一样，变的只是消息走什么管道**。

### 2. stdio 到底是什么

> **关键澄清**：stdio 读作「standard input/output」，是**软件层**的「标准输入输出」，与嵌入式硬件领域的 **SDIO（Secure Digital Input Output，SD 卡总线接口）完全无关**——两者只是拼写相近，不是一回事。本文所有的 stdio 都指前者。

操作系统启动**每个进程**时，会默认给它开三个数据通道（叫「标准流」）：

```
进程启动 → 操作系统自动分配三个文件描述符（fd）：
  fd 0 = stdin  （标准输入）  ← 进程从这里读数据（默认接键盘）
  fd 1 = stdout （标准输出）  ← 进程往这里写正常输出（默认接屏幕）
  fd 2 = stderr （标准错误）  ← 进程往这里写错误信息（默认接屏幕）
```

比如你在终端跑 `node app.js`，这个 node 进程一出生就有这三个通道：它 `console.log()` 往 stdout 写，屏幕上就显示出来；它 `process.stdin.read()` 从 stdin 读，就能拿到你敲键盘的内容。这是 Unix / Linux / Windows **通用**的基础机制，不挑平台。

MCP 的 stdio 传输，就是 client 和 server 之间用 server 进程的 stdin / stdout 这两根管子交换 JSON-RPC 消息：

```
MCP client（agent）                    MCP server（被启动的子进程）
    │                                          │
    │  把 JSON-RPC 请求写进 ──────────────→  stdin  (server 读到请求)
    │                                          │
    │  从读到 ←────────────────────── stdout  (server 写出响应)
    │                                          │
```

MCP 选 stdio 而不是开网络端口的原因：

- **零配置**——不用选端口、不用配防火墙、不用写 IP；
- **生命周期绑定**——client 是父进程，启动 server 作为子进程，client 退出 server 自然跟着死，不会残留；
- **跨平台**——stdin/stdout 任何 OS 都有，Windows / Linux / Mac 一样。

### 3. SSH 怎么把远端的 stdio 搬到本地

这是本文 C1 / C2 / C3 方案的技术底座。当 agent 的 MCP 配置是：

```json
"remote-src": {
  "command": "ssh",
  "args": ["user@build-server", "node /opt/mcp-file-server/cli.js"]
}
```

agent 在本地执行的效果，相当于在终端敲了：

```
ssh user@build-server "node /opt/mcp-file-server/cli.js"
```

发生的事情是 SSH 把**远端进程的 stdin/stdout 接到了本地**：

```
本地 (Windows)                                    远端 (编译服务器)
┌─────────────────┐    SSH 加密隧道     ┌─────────────────────────┐
│ agent (client)  │ ◄──────────────► │ sshd                      │
│                 │                   │   └─ 启动 node mcp-server │
│  往 ssh 写 JSON │ ────网络────────→ │     → 进了它的 stdin      │
│                 │                   │                           │
│  从 ssh 读 JSON │ ←───网络───────── │     ← 来自它的 stdout     │
└─────────────────┘                   └─────────────────────────┘
```

关键点在于 SSH 的一条特性：**它把远端进程的 stdin/stdout 延伸到本地了**。

- 你（本地）往 SSH 这根管子写的数据 → 穿过网络 → 出现在远端进程的 stdin 里；
- 远端进程往 stdout 写的数据 → 穿过网络 → 回到你本地。

所以 agent（在本地）通过 SSH 这根管子，读写的是**远端** MCP server 的 stdin/stdout。对 agent 来说，它以为自己只是在和一个子进程通信（本地有个 stdio 管道）；对远端 server 来说，它以为自己只是在和本地通过 stdio 通信。**中间的 SSH 是透明的搬运工**——它负责把两端的 stdin/stdout 对接起来，两端的进程完全感知不到中间有网络。

这也解释了文档里反复出现的「MCP over SSH 的 stdio 传输」「通过 SSH exec 建立 stdin/stdout 的 JSON-RPC 通道」的全部含义：**数据层不变（还是 JSON-RPC），传输层把本地 stdio 管道用 SSH 隧道延长到了远端**。所谓「远端 stdio」并不是远端有个特殊的 stdio 协议，而是「远端进程的 stdin/stdout 被 SSH 拉到了本地，对 client 透明」。

### 4. SSH 免密

承接上一小节——agent 通过 SSH 拉起远端 MCP server 时，第一个撞上的实操门槛是**鉴权**。这一小节回答：C1 / C2 / C3 要不要免密？为什么？怎么配？

#### 4.1 根因：agent 是非交互进程，没法响应密码提示

MCP client（agent）启动远端 MCP server 的方式，是把它当成一个**子进程**拉起：

```json
"remote-src": {
  "command": "ssh",
  "args": ["user@build-server", "node /opt/mcp-file-server/cli.js"]
}
```

agent 执行这条命令，等价于在终端敲了 `ssh user@build-server "node ..."`。问题在于：

- **如果是密码登录**：SSH 会弹出 `user@build-server's password:` 提示，等待用户从终端输入密码。但 agent 是个**非交互的后台进程**——它的 stdin 是给 JSON-RPC 消息用的管道，**没有人在终端前敲密码**。SSH 等不到输入 → 超时 → 连接失败 → MCP server 起不来 → agent 拿不到这个 server 的工具。
- **如果是公钥免密**：SSH 用本地私钥直接完成认证，不问任何问题，连接瞬间建立，server 立刻起来。

所以不是「MCP 协议要求免密」，而是「**agent 作为非交互进程，没法响应 SSH 的密码提示**」——免密是唯一能让它自动跑起来的方式。

```
agent 拉起子进程: ssh user@build-server "node mcp-server.js"
   │
   ├─ 密码登录: SSH 弹 "password:" → 等 stdin 输入 → 但 stdin 是 JSON-RPC 管道,
   │            没人敲密码 → 超时失败 → MCP server 起不来 ✗
   │
   └─ 公钥免密: SSH 用私钥直接认证 → 不问任何问题 → 连接建立 → server 起来 ✓
```

#### 4.2 三种方案的免密配置

C1 / C2 / C3 的连接方向不同，免密配置的位置也不同：

| | C1 | C2 | C3 |
|---|---|---|---|
| 连接方向 | Windows → 编译服务器（正向） | Windows → 编译服务器（正向） | 编译服务器 → Windows（**反向**） |
| 谁当 SSH client | Windows | Windows | **编译服务器** |
| 谁当 SSH server | 编译服务器 | 编译服务器 | **Windows** |
| 免密配置在哪 | Windows 配私钥，编译服务器加公钥 | 同 C1 | **反过来**：编译服务器配私钥，Windows 加公钥 |
| 是否要求免密 | 实质上是 | 实质上是 | 实质上是 |

C1 和 C2 的免密配置完全一样（方向相同），都是标准的「Windows client → Linux server」公钥免密。C3 是难点——不只是免密，而是整套反向 SSH 基础设施都得搭（见第六节 C3 的 3.2 关键障碍）。

##### 4.2.1 C1 / C2 正向免密配置（Windows → 编译服务器）

标准的公钥免密三步：

```bash
# 1. Windows 上生成密钥对（Git Bash / PowerShell 都行）
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519   # 一路回车，passphrase 留空

# 2. 把公钥推到编译服务器
ssh-copy-id user@build-server
# 或手动: 把 ~/.ssh/id_ed25519.pub 内容追加到编译服务器的 ~/.ssh/authorized_keys

# 3. 验证（不应再问密码）
ssh user@build-server "echo ok"
```

配置完，agent 用 `ssh user@build-server "node ..."` 就能无感拉起远端 MCP。

##### 4.2.2 C3 反向免密配置（编译服务器 → Windows）

C3 要反向，多了两步前置：

（1）**Windows 开 OpenSSH server**（默认未启用）：
- `设置 → 应用 → 可选功能 → 添加功能 → OpenSSH 服务器`；
- 服务里启动 `sshd`，设为自动。

（2）**编译服务器上生成密钥对**，把公钥加到 Windows 的 `C:\Users\<user>\.ssh\authorized_keys`：

```bash
# 在编译服务器上
ssh-keygen -t ed25519
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@windows-host
# 或手动把公钥贴到 Windows 的 authorized_keys
```

（3）验证：在编译服务器上 `ssh user@windows-host "echo ok"`，不应问密码。

C3 的免密配置本身不难，难点在前置的「Windows 开 SSH server + 网络可达性」（见 C3 的 3.2）——这两条是 C3 的真正门槛，免密只是顺带的一步。

#### 4.3 不用免密的替代方案及代价

理论上免密不是唯一出路，但替代方案都引入新麻烦：

| 替代方案 | 怎么做 | 代价 |
|---|---|---|
| `sshpass` | 用 `sshpass -p '密码' ssh ...` 把密码塞进命令行 | 密码明文写在 MCP 配置里，安全差；Windows 上 sshpass 还得额外装 |
| `ssh-agent` | 私钥加进 agent，`SSH_AUTH_SOCK` 环境变量就绪 | 仍需要免密的私钥，只是把「输密码」换成「输 passphrase」；agent 得先手动起 |
| MCP server 预启动 | 不让 agent 拉 SSH，而是手动在远端把 server 跑起来，agent 连已存在的进程 | 破坏了 MCP「client 启动 server」的生命周期绑定，要自己管进程存活 |

这些方案都能绕开免密，但都引入了新的麻烦。**对绝大多数场景，老老实实配公钥免密是最省事的**——一次性配置，之后 agent 每次启动远端 MCP 都无感通过。

## 三、 AI 的文件读写工具：统一语义

在展开各方案前，先厘清一个被下面三节方案（第四、五、六节）跳过的中间层：**AI 实际调用什么工具读写文件，以及这些工具的替换逻辑**。这层搞清楚了，后面每个方案的「读写机制」只需说明数据在哪端流动、过不过网，无需重复解释工具本身。

### 1. AI 用哪个工具？两类来源

AI 的文件操作工具只有两个来源，取决于文件是否在 AI 的本地路径下：

| 来源 | 典型工具名 | 操作对象 | 出现在哪些方案 |
|------|-----------|---------|--------------|
| **agent 内置** | claudecode / zcode / opencode 自带的 `read` / `edit` / `write` | 本地文件系统路径（如 `E:\...` 或挂载盘 `Z:\`） | **A1**（本地副本）、**B3**（挂载盘，对 AI 透明） |
| **MCP server 暴露** | `ssh_file_read/edit/write` 或 `remote_file_read/edit/write` | 远端路径（如 `drivers/net/e1000.c`） | **A2**（ssh_shell_exec）、**B2**、**C1**、**C2** |

- **A1 / B3**：AI 根本不用 MCP 的文件工具，用的是自己**内置**的 `read` / `edit` / `write`——因为文件对它来说就在「本地」（A1 是真本地副本，B3 是挂载出来的假盘，AI 分不清真假）。
- **A2 / B2 / C1 / C2**：远端文件不在 AI 的本地文件系统里，AI 只能调 MCP server 暴露的文件工具，由 MCP server 代为读写。

【**注意**】「用 agent 内置工具还是 MCP 工具」**不是 AI 自己选的，是方案决定的**：文件在不在 AI 的本地路径下，决定了它能不能用内置工具。

### 2. read / edit / write 的标准语义

不管是 agent 内置工具还是 MCP 工具，claudecode / zcode / opencode 这类 agent 的文件工具语义**高度一致**（这是对接同一推理模型的事实标准）。三个工具各自语义如下。

#### 2.1 read：按行读

- 参数：`path` + 可选 `offset`（起始行）+ `limit`（行数）
- 返回：**带行号**的文本
- 语义：**按行**，不是字节；可整文件、可局部行区间

#### 2.2 edit：字符串精确替换（old_str → new_str）

这是理解替换逻辑的关键。答案是：**既不是按行，也不是按字节，是「字符串精确匹配替换」**。

- 参数：`old_str`（要被替换的原文片段）+ `new_str`（新内容）
- 工具在文件里**找到 `old_str` 的精确出现位置**，替换成 `new_str`
- 要求：`old_str` 必须**唯一匹配**（文件里只出现一次），否则报错要求补上下文；有 `replace_all` 选项可关掉唯一性约束
- `old_str` 通常带几行上下文（不是单行），保证唯一

举例：

```
old_str: "    if (ret < 0) {\n        return -EINVAL;\n    }"
new_str: "    if (ret < 0) {\n        dev_err(dev, \"read failed\\n\");\n        return -EINVAL;\n    }"
```

工具找到这块原文、换成新内容，中间插了一行，前后没动。

#### 2.3 write：整个文件全量覆盖

- 参数：`path` + `content`（完整新内容）
- 语义：**直接整个文件全量替换**，原文件被清空、写入新内容
- 用于新建文件或整文件重写

### 3. 关键真相：edit 的「局部替换」是 AI 视角的抽象，底层永远是全量覆盖写

这是理解整个读写机制的钥匙。你问的「按行 / 字节 / 整块 / 全量替换」答案在这里统一：

> **edit 工具的 API 语义是「局部字符串替换」，但底层物理实现永远是「读全文 → 内存里替换 → 全量覆盖写回」。**

原因纯粹是文件系统的物理限制：

- `old_str` 和 `new_str` **长度几乎总是不同**（哪怕差一个字节），替换后**后续所有内容要整体移位**；
- 文件系统**不支持「在文件中间插入 / 删除字节」**，只支持覆盖固定位置的字节；
- 所以唯一的物理实现是：读出全文 → 在内存里拼好新全文 → `O_TRUNC` 清空文件 → 写入完整新内容。

```
AI 视角（edit 局部替换）          文件系统视角（全量覆盖写）
─────────────────────          ──────────────────────────
old_str (50 字节)                 1. read 整个文件 (100KB) 进内存
new_str (80 字节)       ───→      2. 内存里找到位置、拼出新全文
只改了一小块                       3. O_TRUNC 清空文件
省 token                          4. write 整个新全文 (100KB) 落盘
```

【**注意**】

- **AI 的 API 语义**：字符串块替换（`old_str`→`new_str`），不是行、不是字节；
- **磁盘的物理实现**：永远全量覆盖写，没有例外。

「局部」是面向 AI 上下文窗口的省 token 抽象，「全量覆盖」是磁盘的物理事实。两者不矛盾——`edit` 省的是 AI 的 token，没省磁盘的写入量。这也意味着，**后文各方案讨论的「读写机制」差异，不在「替换逻辑」（都一样），而在「这个全量覆盖写发生在哪一端、过不过网、中断风险多大」**。

### 4. 七种方案的工具与替换映射

把工具来源叠加上面的统一语义，各方案的差异点收敛为下表：

| 方案 | AI 用的工具 | 工具来源 | edit 的字符串替换在哪发生 | 底层全量覆盖写落在哪 |
|------|-----------|---------|------------------------|-------------------|
| **A1** 本地副本 | read/edit/write | agent 内置 | **Windows 本地进程内存** | **Windows 本地磁盘**（真副本） |
| **A2** 按需 sed | ssh_shell_exec | MCP（现有） | 不支持写回（只能 `sed -i`，危险） | — |
| **B1** 索引查询 | ssh_shell_exec | MCP（现有） | 不支持写回 | — |
| **B2** MCP 文件层 | ssh_file_read/edit/write | MCP（新增） | **Windows 本地 MCP 内存** | **编译服务器远端磁盘**（过网覆盖写） |
| **B3** 挂载 | read/edit/write | agent 内置（透明） | **Windows 本地进程内存** | **远端磁盘**（过网，挂载层翻译） |
| **C1** 远端 MCP | remote_file_read/edit/write | MCP（远端） | **远端进程内存** | **远端本地磁盘**（零网络） |
| **C2** 双 MCP | remote_file_read/edit/write | MCP（远端） | **远端进程内存** | **远端本地磁盘**（零网络） |

两个最值得注意的对比：

- **A1 vs B2**：AI 都用「字符串替换」语义，但 A1 的替换在本地内存、落在本地副本磁盘（安全）；B2 的替换也在本地内存、却要**过网全量覆盖写**回远端（有中断风险，见 B2 的读写机制小节）。
- **B2 vs C2**：edit 的 API 一模一样（都是 `old_str`→`new_str`），但 B2 替换在本地内存 + 过网写，C2 替换在远端内存 + 本地写。**AI 感知不到差别，token 消耗也一样，差的只是底层网络。**

## 四、 思路 A：把代码搬到本地

核心特征：Windows 本地维护一份代码副本，AI 在本地读、本地改，改完再想办法同步回编译服务器。

### 1. 方案 A1：批量同步到本地（rsync / scp / tar）

#### 1.1 做法

- 在编译服务器把源代码目录暴露为可读；
- 本地通过 `scp -r` / `rsync -avz --exclude=.git` 把整树拉到 Windows 本地工作区（如 `E:\AI\remote-src\<server>`），或者用 `tar czf - <dir>` 打包后传输、本地解压；
- 让 claudecode / opencode 直接打开本地工作区，AI 用本地 Read / Grep / Glob 工具分析，零改动。

```
+-------------------+   rsync/scp/tar  +------------------------+
| 编译服务器          | =============>   | Windows 本地工作区      |
| ~/src/kernel/...  |                  | E:\AI\remote-src\...   |
+-------------------+                  +------------------------+
        ^                                      ^
        |                                      | 本地 Read/Grep/Glob
        |                                      v
        \----------------------------- claudecode / opencode
```

`rsync` 与 `tar` 的选择依据不同：`rsync` 基于**内容差异**做增量同步（只传变化的块），适合反复同步同一仓库；`tar` 是一次性大目录快照，适合首次全量拉取或归档。首次用 `tar` 打包再传，后续迭代用 `rsync` 增量更新，是常见组合。

#### 1.2 优缺点

| 优势 | 局限 |
|------|------|
| 实现最快，不依赖 MCP server 的文件类新工具 | 需要本地磁盘空间（大仓库可能 GB 级） |
| AI 用现成的本地工具链（Read、Grep、Glob），上下文窗口里直接放文件内容、片段、行号引用全部精确 | 远端持续修改时存在「快照陈旧」问题——需要每次开新一轮分析前手动再同步一次 |
| 离线后仍能继续分析，可多次反复 grep | 拉取过程本身有带宽成本，首次同步较慢 |
| 与 git 工作流衔接自然（拉回的就是可 commit 的本地仓库） | `rsync` 适合增量同步、`tar` 适合一次性快照，两者需要用户判断用哪个 |

#### 1.3 读写机制

- **发生在哪一端**：读写都发生在 **Windows 本地**。AI 的 Read / Grep / Glob / Edit 工具操作的是 `E:\AI\remote-src\...` 这份本地副本，编译服务器在整个分析过程中只充当「初始数据源」，同步完成后即不参与读写。
- **内存还是磁盘**：**直接读写本地磁盘**。AI 读文件时，本地工具链经 `read()` 系统调用直接从 Windows 本地磁盘扇区取数据；写文件时直接落盘到本地磁盘。数据不经过 MCP server 进程内存做长期驻留，是真实的本地文件副本（不是挂载的「假盘」，详见思路 B 的方案 B3）。
- **读写粒度**：**可局部、可全量**。本地 Read 支持按行号区间读（局部），Edit 支持局部 patch，Write 支持整文件重写；粒度完全由本地工具链决定，不受网络协议约束。
- **是否全量覆盖**：写回时**可选**——本地 Edit 是局部 patch，本地 Write 是全量覆盖。但这里的「写」只改本地副本，与编译服务器无关。
- **写回编译服务器的额外步骤**：AI 改完本地文件后，需要通过 `rsync` / `scp` 把变更**推回**编译服务器。这不是自动的，必须由 AI 显式调用同步工具，或由用户在编译前手动同步。如果忘记同步，编译的仍然是远端旧代码。存在「本地改完再推回去」的两步工作流和版本一致性问题。

#### 1.4 适用场景

- 远端源码总体稳定、不频繁变动；
- AI 需要做跨文件、反复 grep、整体结构理解这类**重读**任务；
- 本地磁盘和带宽都够用；
- 可以接受「本地改完再推回去」的两步工作流。

### 2. 方案 A2：子目录 / 单文件按需拉取

#### 2.1 做法

- 不全量拉，而是让 AI 通过 `ssh_shell_exec` 执行 `cat` / `head` / `tail` / `sed -n 'a,bp'` 之类的命令，按需把**当前要分析的那一段**内容取回（注意：这是「按行号区间读」，需 AI 自己算行号；与「按内容差异 diff 读」心智模型不同，后者要求远端维护 baseline）；
- 输出会出现在 `ssh_shell_exec` 的返回值里，AI 拿到字符串后即可在上下文中推理。

```
AI -> ssh_shell_exec(session, "sed -n '120,180p' drivers/net/e1000.c")
       |
       v
SSH 通道 -> 远端 sed -> 返回那段文本
       |
       v
AI 上下文里得到那段代码 -> 继续推理
```

每次取文件的标准流程是「先想读哪段 → 发命令 → 等回显」三步往返，延迟逐次累加。长会话还要注意 `MAX_BUFFER_SIZE` 约束——大文件 `cat` 时输出会被切，需要在 MCP server 侧或 AI 侧做净化（去 ANSI 颜色码、合并 `\r\n`）。

#### 2.2 优缺点

| 优势 | 局限 |
|------|------|
| **零本地存储**，磁盘压力小 | 每个文件都要三步往返，延迟累加 |
| **零部署**，完全用现有 MCP server 的 `ssh_shell_exec` 工具就能干 | 命令行输出会被终端宽度截断、被转义字符污染（颜色码 / `\r`）——需净化 |
| 每次只取真正相关的那一段，上下文窗口利用率高 | 没有全局视图，AI 难以知道仓库全貌，得靠 `ls` / `find` 边走边探 |
| - | 长会话容易触发 SSH 输出缓冲的截断（受 `MAX_BUFFER_SIZE` 约束，会在大文件 cat 时被切） |

#### 2.3 读写机制

- **发生在哪一端**：读发生在 **编译服务器远端**。`ssh_shell_exec` 在远端执行 shell 命令（`cat` / `sed -n`），远端进程读远端磁盘文件，把文本通过 SSH 通道回传给 Windows 本地的 MCP server，MCP server 再把字符串放进返回值交给 AI。
- **内存还是磁盘**：**回显到内存，不落本地磁盘**。取回的文件内容以字符串形式存在于 `ssh_shell_exec` 的返回值里，AI 拿到后只在上下文窗口（内存）中推理。本地磁盘上不会产生任何文件副本——这是 A2 与 A1 的根本区别。
- **读写粒度**：**读局部（按行号区间）**。通过 `sed -n 'a,bp'` 指定行号，只取关心的那一段。粒度由命令行参数决定，最小可到单行。
- **是否全量覆盖**：**写回不支持，或极不推荐**。`ssh_shell_exec` 本质上是跑命令，AI 要改文件只能再发一条 `sed -i` 或 `echo > file` 命令，风险高、不可预测、没有回滚能力。因此该方案只适合只读分析，不适合 ch02 的「AI 改代码 → 编译」闭环。

#### 2.4 适用场景

- 已经定位到目标文件、只想深入读几段，不需要全局漫游；
- 仓库极大、本地不便整拉；
- 临时排查 bug、看少数关键函数；
- **只读场景**，不需要 AI 修改代码。

## 五、 思路 B：把 AI 的读写能力延伸到远端

核心特征：代码仍然留在编译服务器上，AI 通过 MCP tools 直接读写远端文件，本地不维护或只维护临时副本。

### 1. 方案 B1：远端索引与搜索查询（cscope / ripgrep）— 只读辅助

#### 1.1 做法

- 在编译服务器上预跑 `cscope -b -R`、`ctags -R`、`universal-ctags` 建索引；
- AI 通过 `ssh_shell_exec` 调用 `cscope -L1 <symbol>` / `rg <pattern>` / `grep -rn` 来查「谁调用了我」「定义在哪」「这个宏被哪些文件用」；
- 只回传结构化的查询结果，不传源代码正文。

```
AI -> ssh_shell_exec: "cscope -L1 e1000_transfer"
       |
       v
远端 cscope 查询本地索引
       |
       v
返回 "(file.c:line) call site 1\n..."
       |
       v
AI 在上下文里拿到调用关系, 再按需 sed -n 读具体行
```

工作流是「查索引 → 读文件」两段：B1 负责定位（返回 file:line 列表），真要看代码片段仍需 A2 / B2 配合读取。

#### 1.2 优缺点

| 优势 | 局限 |
|------|------|
| **查询本身极快**，远端 cscope 索引在内存里，调 k 行结果秒回 | 依赖编译服务器有 `cscope/ctags/ripgrep`，且索引要预先建好（首次较慢） |
| 网络流量极小，只传查询结果不传源码 | **索引陈旧**：远端源码改动后要重新 `cscope -b`，否则结果不匹配 |
| 适合做大型代码库的「跨文件符号关系」分析，补 A2 的短板 | AI 只能拿查询结果，真要看代码片段仍需 A2/B2 配合 |

#### 1.3 读写机制

- **发生在哪一端**：查询发生在 **编译服务器远端**。cscope / ripgrep 在远端读远端本地索引和远端磁盘文件，只把匹配结果（`file:line` 列表）回传。
- **内存还是磁盘**：**查询走远端内存索引**。cscope 建好的索引常驻远端内存，查询是内存级别的查找；回传的是结构化文本，不落本地磁盘。
- **读写粒度**：**不读正文**，只返回定位信息（`file.c:line`）。要看正文需配合 A2 / B2。
- **是否全量覆盖**：**不支持写回**。B1 只能搜索和定位，本身不提供文件修改能力。它可以作为 B2 / C1 的辅助（先搜索再修改），但不能独立完成「改代码 → 编译」闭环。

#### 1.4 适用场景

- 仓库巨大（如整个 Linux 内核、整个 Android 框架），不可能整拉到本地；
- AI 任务以「定位、调用链、依赖分析」为主；
- 编译服务器本身就是开发主战场，索引长期维护；
- 配合 B2 / C1 使用，先搜索再修改。

### 2. 方案 B2：MCP 文件工具层（推荐演进方向）

#### 2.1 做法

给当前 MCP server 新增一组语义化文件工具，把 `ssh_shell_exec` + 一堆 shell 命令的细节封装成统一的「读写远端文件」接口：

| 新工具名 | 等价远端操作 | 作用 |
|---------|------------|------|
| `ssh_file_read` | `cat` / `sed -n 'a,bp'` / SFTP readFile | 按路径 + 可选行号区间读远端文件（局部读，非内容 diff） |
| `ssh_file_write` | SFTP writeFile / 原子写 | **全量覆盖写**——用于新建文件或整文件重写 |
| `ssh_file_edit` | SFTP 局部写 + 行号/oldStr 匹配 | **局部 patch 写**（即第三章 2.2 所述 edit 工具的 `old_str`→`new_str` 局部替换）——ch02「AI 改代码」主力写工具，详见 2.3 |
| `ssh_file_list` | `find` / `ls -la` / SFTP readdir | 列目录树 |
| `ssh_file_grep` | `rg --json` / `grep -rn` | 远端搜索，回传结构化匹配 |
| `ssh_file_stat` | `stat` / SFTP stat | 拿 mtime / size，用于判断是否变化 |

底层可以有两种实现方式：

##### 2.1.1 方式一：shell 命令翻译

AI 给 path + lineRange，MCP server 翻译成 `sed -n 'a,bp'` 等命令执行，再净化输出。实现快，但受终端输出、ANSI 码、`MAX_BUFFER_SIZE` 限制。

##### 2.1.2 方式二：SFTP 结构化访问

> 【**SFTP 是什么**】
>
> SFTP（SSH File Transfer Protocol）是 SSH 的一个子系统，**跑在已有 SSH 连接之上，不是独立协议**，也和传统 FTP 没有关系。一条 SSH 连接能开多条通道，shell、exec、sftp 各占一条：
>
> ```
> SSH 连接(端口 22, 加密隧道)
>   ├── shell  通道  ← ssh_shell_open 用这个
>   ├── exec   通道  ← ssh_shell_exec 用这个
>   └── sftp   通道  ← B2 的文件工具用这个
>          └── sftp-server 进程(远端负责真正读写磁盘)
> ```
>
> SFTP 的特点是**不只能传文件内容**，还能发结构化命令（读 / 写 / 列目录 / 查属性 / 重命名 / 删除），是一套完整的文件操作原语。所以 B2 能用 SFTP 实现 `readFile` / `writeFile` / `readdir` / `stat` 这一整套。
>
> 【**需要额外安装吗**】
>
> 对 ch02 场景，SFTP **不需要任何额外安装**：
>
> - **远端**：`sftp-server` 随 OpenSSH server 一起打包，几乎总是默认开启。只要编译服务器现在能 SSH 登录，SFTP 就几乎肯定可用——两者是同一个 `sshd` 进程提供的。
> - **本地**：项目用的 Node.js `ssh2` 库内置 SFTP 客户端能力，调 `.sftp()` 即可开通道，不需要额外装 SFTP 客户端程序，零新增依赖。
>
> 这也是 B2「优先 SFTP 实现」在部署上的隐性优势——复用现有 SSH 连接、不开新端口、远端零安装。

MCP server 通过 `ssh2` 打开 SFTP 子通道，直接调用 `readFile` / `writeFile` / `readdir` / `stat` 等原语。文件内容以二进制安全方式传输，不走 pty/shell，输出干净、稳定。

```
AI -> ssh_file_read(path="drivers/net/e1000.c", start=120, end=180)
       |
       v
MCP server 内部:
  ├─ shell 方式: ssh exec "sed -n '120,180p' drivers/net/e1000.c"
  └─ SFTP 方式: sftp.readFile("drivers/net/e1000.c") 后按行切片
       |
       v
返回干净文本(带行号)
       |
       v
AI 上下文塞进去继续推理
```

#### 2.2 优缺点

| 优势 | 局限 |
|------|------|
| AI 不需要懂 shell 命令细节，**调用面干净**——传 path + lineRange 即可 | **需要开发**，不是零成本方案 |
| 输出经 MCP server 统一净化（去 ANSI、合并 `\r\n`），避免 A2 的转义污染陷阱 | 工具粒度与命令性能之间存在权衡（要不要支持 glob？要不要支持 follow symlink？） |
| 可以做分块流式返回（大文件分块），缓解 `MAX_BUFFER_SIZE` 截断问题 | 文件级状态（mtime 缓存等）需要会话级管理 |
| SFTP 方式复用 SSH 连接、协议标准、单文件读写稳定，比 sshfs 性能好 | SFTP 方式大量小文件随机访问时仍有 RTT 累积，批量同步不如 rsync |
| 同样的接口形态可以**并行**支持本地（直接 fs）/ 远程/容器多种后端，对 AI 一致 | - |

#### 2.3 读写机制

本节是 B2 的核心，详细回答四个问题。

##### 2.3.1 发生在哪一端

**MCP server 跑在 Windows 本地，文件本体在编译服务器远端。** 读写操作的「发起方」在本地、「数据所在」在远端，中间隔着 SSH 隧道。这是 B2 与 C1 / C2 的本质区别——后者的 MCP server 本身就跑在远端。

写回分两种粒度，需按场景选择：

| 写工具 | 粒度 | 适用场景 | 风险/约束 |
|--------|------|---------|----------|
| `ssh_file_write` | 全量覆盖 | 新建文件、整文件重写 | AI 易在重写时漏行；无 baseline 校验，会静默覆盖远端并发改动 |
| `ssh_file_edit` | 局部 patch | AI 修改大文件中的几处（ch02 主力） | 省 token；可带 baseline（oldStr / 行号快照）做乐观锁，远端已变则拒绝写 |

- **全量写 vs patch 写的权衡**：几千行的 C 源文件，全量读全量写每次都吃掉大量上下文窗口，AI 还容易丢内容；patch 写只需保证片段正确，且能通过 baseline 校验防止「AI 读 t1 快照、远端已到 t2」的静默覆盖问题。
- `ssh_file_edit` 的参数建议为 `path + (lineRange | oldStr) + newContent + optional baseline`，语义对齐本地 edit 工具（`old_str→new_str`）。

##### 2.3.2 「假局部」：局部读写 API 对 AI 友好，底层仍传全量文件

B2 的工具 API 可以设计得对 AI 很友好——支持 `remote_file_read(path, offset, limit)` 按行读、`remote_file_edit(path, old_str, new_str)` 局部 patch。对 AI 来说 token 消耗相同，但**底层网络传输量完全不同**：

| 操作 | AI 视角（省 token） | B2 底层网络 |
|------|-------------------|------------|
| 局部读 20 行 | 一样，只收 20 行 | **传整个文件** |
| patch 改 1 行 | 一样，只发 old_str/new_str | **传整个文件 ×2**（读+写） |
| 全量读 | 一样 | 传整个文件 |
| 全量写 | 一样 | 传整个文件 |

【**根因：SFTP 协议的字节语义 vs AI 需要的行/字符串语义之间的鸿沟。**】

SFTP 协议本身支持 `SSH_FXP_READ(handle, offset, length)` 字节级局部传输，并非只能全量。但 AI 要的是「读第 40-60 行」或「把 old_str 替换成 new_str」，这两个能力在 SFTP 下都会退化成全量传输：

**（1）按行号读：行号→字节偏移的映射需要先拉全文**

假设 AI 调用 `ssh_file_read("main.c", offset=40, limit=20)`（读第 40-59 行，共 20 行），从远端磁盘到 AI 上下文的完整数据流如下：

```
编译服务器(远端磁盘)              Windows 本地(MCP server 进程)           AI 上下文
─────────────────              ─────────────────────────────         ─────────
main.c 共 3000 行, ~100KB

sftp.readFile("main.c")
  ←──── 整个文件 100KB 字节流 ────  收进进程内存
  (过 SSH 隧道传输)                拿到完整 3000 行的原始字节
                                  按 \n 切行, 取第 40-59 行(20 行)
                                  加行号格式化
                                  其余 2980 行 ──→ 丢弃(不进上下文)

                                                                   只收到这 20 行:
                                                                   40	xxx
                                                                   41	yyy
                                                                   ...
                                                                   59	zzz
```

关键在于**网络上跑的和 AI 看到的不是同一份东西**：

- **网络上跑的**：整个文件（~100KB）。不管 AI 只要 20 行，SFTP 的 `readFile` 都先把整个文件从远端经 SSH 隧道拉到 Windows 本地 MCP server 的进程内存里。
- **AI 看到的**：只有那 20 行，**带行号**的文本（`40\txxx` 这种格式，和本地 read 工具输出一致）。其余 2980 行的原始字节在 MCP server 本地内存里切完就丢弃了，**不进 AI 上下文窗口**。

`offset` / `limit` 这两个局部参数，**省的是 AI 的上下文窗口（token），没省网络往返的数据量**。

根因是 SFTP 的字节语义和 AI 的行语义之间的鸿沟：SFTP 只认字节偏移，不认行号。要知道第 40 行的字节位置，得先知道前 39 行各有多长——这需要先拉一次全文。除非远端有常驻进程维护行偏移索引（那已经不是「纯 SFTP」了，复杂度逼近 C2）。

**（2）patch 写：替换发生在本地，SFTP 只负责搬运**

关键认知：**B2 的 MCP server 跑在 Windows 本地，字符串替换发生在本地内存中，远端的 sftp-server 只负责读写字节，不做任何内容处理。** SFTP 是个「搬运工」——把文件从远端搬到本地，改完再搬回去：

```
Windows 本地（MCP server）                    编译服务器

sftp.readFile("main.c")  ──────网络──────→  sftp-server 读磁盘
                          ←─────网络──────  返回整个文件内容

本地内存中做字符串替换
old_str → new_str

sftp.writeFile("main.c", 新全文) ──网络──→  sftp-server 清空+写磁盘
                                ←──网络──  返回成功
```

具体到一次 patch 操作：

```
AI 要求：remote_file_edit("main.c", old_str="AAA", new_str="BBBB")

B2 底层实际发生：
  sftp.readFile("main.c")         → 全文拉过来（网络传整个文件）
  本地做字符串替换 AAA → BBBB        → 本地修改（在 Windows 内存中）
  sftp.writeFile("main.c", 全文)   → 全文写回去（网络再传整个文件）
```

old_str（3 字节）比 new_str（4 字节）短，替换后后续所有内容要后移一个字节。SFTP 的 `SSH_FXP_WRITE` 是覆盖写，不会自动后移后续字节。所以 patch 写的标准做法是读全文 → 本地替换 → 写回全文。

这也解释了「假局部」的根因：AI 只发了 `old_str` + `new_str`（~50 字节的 patch），但 B2 底层把整个文件拉到本地、改完再推回去（~100KB 网络传输）。网络上传的是整个文件 ×2，不是那 50 字节。

**（3）实际使用的 ssh2 库更是全量 API**

项目使用的 Node.js `ssh2` 库，其 SFTP 封装 `sftp.readFile()` / `sftp.writeFile()` 本身就是全量读写。虽然 `sftp.createReadStream(path, { start, end })` 支持字节范围读，但只解决「按字节读」，不解决「按行读」。实际实现局部读写，基本都会退化成「读全文 → 本地裁剪/替换 → 写回全文」。

**（4）这意味着什么**

B2 的局部读写设计**仍然值得做**——它省 token，避免上下文窗口爆炸，这是比网络延迟更硬的约束。但要注意：

- B2 的 `ssh_file_read` / `ssh_file_edit` 省的是 token，**不是网络**——读时底层传整个文件，写时传整个文件 ×2（读一次 + 写一次）；
- C2 的 `remote_file_read` / `remote_file_edit` 既省 token **又省网络**——`fs.readFile` / `fs.writeFile` 是远端本地调用，读时网络上只回传那几行、写时只传 old_str/new_str 和结果；
- 同样读第 40-59 行，B2 传了整个文件（~100KB）过网，C2 只传那 20 行（~1KB）过网——AI 看到的内容一模一样，差的全部在底层网络；
- AI 改 10 个文件，B2 底层传了 20 次完整文件内容过网，C2 全是本地操作——这是性能差距的一个隐藏放大因素。

##### 2.3.3 内存还是磁盘

**替换在 Windows 本地内存完成，最终落编译服务器磁盘。** 以 patch 写为例：MCP server 通过 SFTP 把远端文件全文读到本地进程内存 → 在内存中做字符串替换 → 把替换后的完整内容通过 SFTP 写回远端磁盘。本地不落临时文件（除非用原子写，见 2.3.4），远端文件在写回时被覆盖。

读操作同理：SFTP 把远端文件字节读进 MCP server 进程内存，按行切片裁剪后，只把目标片段放进返回值交给 AI。整个过程数据在内存中流转，不落本地磁盘。

> 【**「进程内存」精确指什么**】
>
> 上文说的「进程内存」不是含糊的说法，也不是某个临时文件，而是 **Node.js 的 `Buffer` 对象，分配在 MCP server 这个 Node.js 进程的 V8 堆内存里**。ssh2 库的 `sftp.readFile(path, cb)` 回调返回的就是一个 `Buffer`，这是文件内容在本地这一侧的最终落脚点。
>
> 从远端磁盘到本地 Buffer 的完整搬运链路，注意区分哪里是内存、哪里是磁盘：
>
> ```
> 编译服务器远端                              Windows 本地
> ─────────────                              ─────────────
> [磁盘] main.c 原文件
>    ↓ sftp-server 进程 read() 系统调用
> [远端进程内存] sftp-server 的读缓冲
>    ↓ SSH 隧道传输(网络字节流)
>    ↓                                          ↓
> [内核 socket 缓冲]                       [内核 socket 缓冲]   ← 临时,瞬态
>                                               ↓ ssh2 库从 socket 读出
>                                            [MCP 进程堆内存] Node.js Buffer  ← 最终落脚点
>                                               ↓ 按 \n 切行、取目标行
>                                            [MCP 进程堆内存] 返回给 AI 的字符串
>                                               ↓ 用完后无引用 → GC 回收
> ```
>
> 三个要点：
>
> - **全程不落本地磁盘**——从远端磁盘读出后，数据一直在内存里流动（远端进程内存 → 网络字节流 → 内核 socket 缓冲 → ssh2 的 Buffer → 返回字符串），本地磁盘上自始至终没有产生任何文件。这也是 B2 与 A1 的根本区别：A1（rsync）在本地磁盘落真实副本文件，B2 只在内存里临时放一下。
> - **中间的内核 socket 缓冲是瞬态的**——数据穿过 TCP/SSH 隧道时确实会进两端内核的 socket 缓冲区，但那是传输途中的过路内存，流完即空，不是落脚点。代码能看到、能引用的最终结果就是那个 Buffer。
> - **用完即被 GC 回收**——MCP server 拿到 Buffer → 切行取目标行 → 返回结果给 AI 后，这个 Buffer 无人引用，Node.js 的垃圾回收器自动回收这块堆内存。它是**临时、一次性**的内存占用，不持久驻留。
>
> 这也带来一个隐性压力：读一个 100KB 的源文件，MCP server 堆里就临时多 100KB，一般无感；但如果 AI 去读几十 MB 的生成文件（编译产物、log），这块内存占用会很明显——这正是 B2「分块流式返回、缓解 `MAX_BUFFER_SIZE` 截断」要解决的问题。

##### 2.3.4 是否全量覆盖：是，且有中断风险

**（1）patch 写的底层是全量覆盖写，不是局部插入**

不管是全量写还是 patch 写，底层写文件都是同一个流程：

```
1. 读出整个文件的当前内容       → "AAA...old_str...BBB"
2. 在内存里做字符串替换         → "AAA...new_str...BBB"
3. 把替换后的完整内容写回文件    → O_TRUNC 清空文件 + 写入全部新内容
```

第 3 步是全量覆盖写——文件被清空（`O_TRUNC`），然后写入整个修改后的内容。不是只改 old_str 那几个字节。原因很简单：old_str 和 new_str 长度往往不同，替换后后续所有内容要移位，而文件系统不支持「在文件中间插入字节」，只能读全文 → 内存替换 → 全量覆盖写回。

**（2）全量覆盖写的中断风险：B2 的网络窗口很危险**

可怕的不是覆盖写本身，而是「清空后写到一半被打断」。B2 的 SFTP 写入窗口很长：

```
B2（SFTP writeFile）：
  SFTP OPEN(O_TRUNC)    ← 远端文件被清空，~30ms（网络往返）
  SFTP WRITE chunk 1    ← 传第一个块，~30ms
  SFTP WRITE chunk 2    ← 传第二个块，~30ms
  ...
  SFTP CLOSE            ← ~30ms
  整个窗口：~60ms 到数秒（取决于文件大小）
```

而中断的原因在网络场景里是「日常事件」：

| 中断类型 | B2（网络） |
|---------|-----------|
| 网络抖动 | **常见**（WiFi 波动、VPN 掉线） |
| SSH 会话超时 | **常见**（空闲超时、服务器 kill） |
| 服务器重启 | 偶发 |

B2 如果在 `O_TRUNC` 之后、写入完成之前连接断开，**文件已被清空但只写了一部分**，一个 3000 行的源文件变成几百行残缺内容——这是数据损坏，不是简单的「写失败重来」。

**（3）原子写：临时文件 + rename**

要解决覆盖写的中断风险，标准做法是先写临时文件再原子替换：

```ts
// B2 安全写法
sftp.writeFile("main.c.tmp", newContent) // 写临时文件，过网络
sftp.rename("main.c.tmp", "main.c")      // 原子替换，过网络
```

`rename` 是原子操作——文件系统保证要么看到旧文件、要么看到新文件，不会看到「半个」。写入过程都在临时文件上进行，原文件不受影响；临时文件写完才 rename 过去。

关键差异：

- B2 **必须写临时文件 + rename**才安全，否则网络断开时原文件可能被清空到一半；
- 但 ssh2 库的 `sftp.writeFile()` 默认直接覆盖原文件，**不做临时文件保护**——要安全写法得自己实现；
- B2 用了临时文件后，可靠性大幅提升——最坏情况是「写失败，原文件没变」（临时文件残缺但没 rename），不是「写失败，原文件损坏」。

##### 2.3.5 模糊失败：B2 独有的隐患

除了文件损坏，B2 还有一个 C2 没有的问题——**写入确认可能丢失**：

```
数据已到达远端并写入磁盘
  ↓
远端回复「写入成功」
  ↓
回复在传输途中丢失（连接断开）  ← 这里断了
  ↓
本地 MCP 收不到成功确认 → 不知道文件到底写了没有
```

B2 的写入确认要过网络回来，如果回复丢失，MCP 面临两难：

- 告 AI「写入失败」→ 但文件其实已改，AI 重试可能导致重复操作；
- 告 AI「写入成功」→ 但可能没写，AI 基于错误假设继续工作。

**模糊失败是分布式系统里最难处理的故障类型**，B2 天然带有这个风险，C2 没有。

#### 2.4 可扩展的高级工具

B2 的抽象层之上，未来还可以叠加一些高级工具，但它们不是独立方案，只是 B2 工具集的扩展：

| 高级工具 | 作用 | 与方案的关系 |
|---------|------|------------|
| `ssh_file_summarize` | 对单个文件或目录生成摘要（函数签名、关键注释、依赖列表） | 原作为独立摘要方案的能力，现合并为 B2 的一个 tool |
| `ssh_file_index` | 调用远端 `cscope` / `ctags` 建索引，或维护项目级符号索引 | 替代/增强原「B1 远端索引查询」，AI 先搜索引再读文件 |
| `ssh_file_diff` | 比较远端文件与本地/历史版本差异（**只读比较**，不是 patch 写回） | 辅助 AI 判断修改范围，与 `ssh_file_edit` 的写操作区分 |

这些工具可以按需实现，不影响 B2 的核心架构。

#### 2.5 适用场景

- 长期、反复用 AI 分析并修改远端源码；
- 希望一次开发、多种 transport（SSH / ADB / Serial）共享同一文件抽象；
- 是 ch02 最值得推进的方向之一。

### 3. 方案 B3：远端文件系统挂载（sshfs / rclone / SMB）

#### 3.1 做法

- 在 Windows 上用 `sshfs-win` / `rclone mount` / WinSCP / SMB（`net use`）挂载编译服务器的源码目录到本地盘符（如 `Z:\remote-src\`）；
- 让 claudecode / opencode 直接像访问本地文件一样 Read / Grep / Write 它。

对 AI 来说挂载盘和本地盘没区别，工具链零改动；但底层数据仍在远端，每次 I/O 都要过网。

#### 3.2 优缺点

| 优势 | 局限 |
|------|------|
| 对 AI **完全透明**——它根本不知道文件在远端，工具链零改动 | Windows 上 `sshfs-win` / WinFsp 稳定性一般，长会话可能掉盘 |
| 实时反映远端改动（挂载语义是「按需读」），不会像 A1 那样陈旧 | 每次 Read 都要触发一次网络往返，**搜索（grep）极其痛苦**——每次匹配都是一次网络 IO，远比 A1 慢 |
| 不占本地磁盘空间（按需 fetch） | 不适合 AI 反复读同一文件、来回 grep 的工作模式（延迟累计） |
| 写操作自动同步回远端 | - |

##### 3.2.1 为什么挂载方案搜索性能极差

【**注意**】

> **关键认知**：挂载 ≠ 复制。挂载出来的盘符（如 `Z:\`）只是一个「访问入口」，**数据本体仍在编译服务器的磁盘上**，本地没有任何真实副本。挂载动作只注册了一个虚拟文件系统，并没有把文件落到本地磁盘——后者是 A1（rsync 全量同步）干的事。

**（1）挂载盘是「会自动联网取数据的假盘」**

sshfs 建立在 FUSE（Filesystem in Userspace）之上。AI 的 read/grep 工具底层走的是标准 `read()` 系统调用，工具链不知道也不需要知道 `Z:\` 是挂载盘。内核拿到请求后的路径解析如下：

```
read(Z:\remote-src\foo.c)
  ↓ 内核解析路径
  ↓ 发现 Z:\ 是 FUSE 挂载点，不是真实磁盘
  ↓ 把请求交给 FUSE 框架
  ↓ FUSE 转交给 sshfs 用户态进程
  ↓ sshfs 翻译成 SFTP READ 请求
  ↓ 走 SSH 隧道到编译服务器
  ↓ 远端 sftp-server 读真实磁盘
  ↓ 原路返回文件内容
```

内核手里没有这个文件的数据，它只知道「这个路径归 FUSE 管」，于是每次都得让 sshfs 去远端现取。内核不会、也无法把文件缓存成「本地真实数据」——FUSE 层面虽有页缓存，但那是临时缓存、会失效，不是磁盘上的真实副本。

**（2）read 单文件 vs grep 的区别只在「次数」**

单个文件 read 一次 = 一次 SFTP 往返 ≈ 几十毫秒，AI 基本无感，这也是 sshfs 在「偶发看一两个文件」场景还能用的原因。grep 崩溃在于它是 read 的**几千倍放大**：

- 遍历目录树要 `readdir`（每次也是网络往返）；
- 每个文件要 `open` + 多次 `read` + `close`（又是多次往返）；
- 这些往返基本串行，延迟随文件数量**线性累加**——上万文件 × 几次往返 × 几十毫秒 = 分钟级，而本地盘 grep 是秒级。

文档结论即「每次匹配都是一次网络 IO，远比 A1 慢」——A1 的数据已在本地磁盘上，grep 直接扫本地扇区；sshfs 的数据还在远端，grep 每碰一个文件都要过一次网。

##### 3.2.2 SMB 呢？比 sshfs 强，但搜索仍是瓶颈

SMB（Windows 原生网络文件系统协议，服务端常由 Samba 提供）是所有「网络挂载」方案里体验最好的一个，但它改善的是**每次往返的效率**，没有改变「数据本体仍在远端、本地没有真实副本」这个根本事实。

**（1）SMB 在协议/实现层面优于 sshfs**

| 维度 | sshfs (FUSE + SFTP) | SMB (Windows 内核 redirector) |
|------|---------------------|-------------------------------|
| 实现位置 | **用户态**（FUSE 框架） | **内核态**（Windows 原生驱动） |
| 协议设计 | SFTP，一个操作一个请求 | 支持 **compound 请求**（多个操作打包一次往返） |
| 目录列举 | 逐条 readdir 往返 | `FindFirst/FindNext` 批量返回 |
| 缓存 | FUSE 页缓存，有限且受实现限制 | 内核级 **oplock/lease** + 目录缓存 + 文件数据缓存 |
| 单次往返延迟 | 用户态切换 + SFTP 解码，较高 | 内核态直通，较低 |

日常用 Windows 映射网络驱动器（`net use Z: \\server\share`）挂 SMB，打开文件、编辑的感觉会比 sshfs 顺滑很多。

**（2）但 grep 全量搜索仍然会痛**

SMB 改善的是「单次往返的效率」，无法消除「数据不在本地、每次 I/O 都要过网」这个架构性问题。grep 第一次全量扫描一个项目时：

1. **缓存全是冷的**——成千上万个文件，每一个都是第一次读，oplock/lease 缓存帮不上忙；
2. **内容仍要过网**——每个文件的字节都得从远端磁盘取过来，SMB 只是把「取」做得更快、更批量，但不能让数据凭空出现在本地；
3. **延迟仍然线性累加**——一万个文件，哪怕每个只需 5ms（比 sshfs 的 30ms 快很多），也是 50 秒，而本地盘 grep 是 1 秒以内。

第二次 grep 同一批文件会快很多（缓存热了），但 AI 的工作模式经常是改了文件后再 grep、搜索新关键词、探索不同目录——缓存命中率并不稳定。

**（3）本质对比：挂载类方案的天花板**

```
sshfs 挂载   → 数据远端，用户态翻译，每次 I/O 过网，且单次慢   → 搜索极痛
SMB 挂载     → 数据远端，内核态协议，每次 I/O 过网，但单次快   → 搜索仍痛，但可忍
本地副本(A1) → 数据本地，直接扫磁盘扇区                       → 搜索快，但要同步
Remote-SSH   → 计算搬到数据旁，grep 在远端本地跑，只回传结果   → 搜索快，无需同步
```

SMB 解决的是「单次 I/O 过网太慢」，但解决不了「搜索要批量 I/O 且数据不在本地」这个架构性问题。换 SMB 后痛苦程度从「极痛」降到「可忍但仍是瓶颈」，B3 方案「搜索极其痛苦、不推荐用于 AI 重分析」的结论方向不变。

#### 3.3 读写机制

- **发生在哪一端**：**对 AI 透明**——AI 以为在操作本地盘 `Z:\`，但数据本体仍在**编译服务器远端**。每次 read/write 经挂载层翻译成 SFTP / SMB 请求，过 SSH 隧道或网络协议到达远端。
- **内存还是磁盘**：**每次过网落远端磁盘**。挂载盘不是真实本地副本（区别于 A1），数据在远端磁盘上，本地内核无数据。读时现取、写时现推。FUSE / SMB 层有页缓存，但那是临时缓存、会失效，不是磁盘上的真实副本。
- **读写粒度**：**按需局部**。AI 可用本地 Edit 做局部 patch、Read 按行区间读，粒度由本地工具链决定；但底层每次操作（无论粒度多小）都要触发一次网络往返。
- **是否全量覆盖**：**写自动同步回远端**，但写中断风险同 B2（过网覆盖写）。AI 修改本地挂载盘中的文件，底层会自动通过 SFTP / SMB 写回编译服务器。对 AI 来说和改本地文件没有区别，但搜索性能是硬伤（见 3.2.1）。

#### 3.4 适用场景

- 文件量小、读次数少、要严格保持实时；
- 不是 AI 重度分析场景，AI 只偶发看/改一两个文件——但偶发场景用 A2/B2 更轻；
- **不推荐用于 AI 重分析**：原因即上述搜索性能问题，正是 [`vscode-sshremote.md`](vscode-sshremote.md) 第八节对比里讨论过的「挂载网络盘」痛点。

## 六、 思路 C：远端跑一个服务

核心特征：在编译服务器上跑一个完整或精简的服务，AI 通过网络协议远程接入，所有操作在服务器本地完成。

### 1. 方案 C1：远端部署 MCP server（JSON-RPC 远程服务进程）

#### 1.1 做法

- 把现在的 MCP server 直接跑到编译服务器上（Linux 版本），以 stdio 方式运行；
- 本地 claudecode / opencode 通过 SSH exec 建立 stdin/stdout 的 JSON-RPC 通道远端接入（不开远程端口），所有文件工具直接命中远端本地文件系统——**就是 VSCode Remote-SSH 的「反向」版本**。

```
+---------- Windows ----------+                       +----- 编译服务器 ---------+
| claudecode/opencode         | MCP over SSH/JSON-RPC | MCP server (Linux)     |
|   |  远端 MCP 客户端          | <==================>  | file tools (本地 fs)   |
+-----------------------------+                       +------------------------+
                                                              |
                                                              v
                                                       [远端本地磁盘]
                                                       [远端 cscope / rg]
                                                       [远端 build / git]
```

#### 1.2 优缺点

| 优势 | 局限 |
|------|------|
| 体验最接近 VSCode Remote-SSH 的「无感」——文件读写、grep、构建都在远端就地完成 | 需要把 MCP server 移植 / 部署到 Linux，要解决跨平台编译、Node 依赖、IPC 文件路径差异 |
| AI 看到的就是远端**实时**状态，无任何同步开销 | 鉴权依托 SSH 账号体系（走 SSH exec 通道不开远程端口，比裸跑 TCP/SSE 安全），但仍需注意 SSH 账号权限收敛 |
| 与 ch02 长远目标「让远端 AI 流程能反向访问本地设备」天然对齐——可在远端 MCP server 上再加「本地设备代理」工具把本地 USB /串口反向暴露出去 | 仍然是「AI 不会把源码搬到本地」，**远端跑挂 = AI 失明**，没有离线能力 |
| - | 当前 MCP server 架构是面向本地（Windows 主机）的，移到远端后 PowerShell 工具、serial 工具、port_scan 等就失去意义，要做裁剪 |

#### 1.3 读写机制

- **发生在哪一端**：**全部在编译服务器远端**。MCP server 本身跑在远端，文件工具直接命中远端本地文件系统（`fs.readFile` / `fs.writeFile`），数据不经网络传输。
- **内存还是磁盘**：**直接读写远端本地磁盘**。远端 MCP server 进程直接操作远端文件系统，零网络 I/O——这是 C1 / C2 与 B2 的根本区别。
- **读写粒度**：**读局部 / 写局部或全量**（edit + write），粒度由远端 MCP server 的工具实现决定。
- **是否全量覆盖**：**支持，直接写回远端**。AI 调用远端 MCP server 的文件 tools 修改文件，修改直接发生在编译服务器本地。写中断风险等同于普通本地 fs 写（见 C2 的 2.3），远低于 B2 的网络覆盖写风险。这是最强的写回之一。

#### 1.4 适用场景

- 长期、合作式开发场景——编译服务器是主战场，AI 反复分析与修改同一仓库；
- 已经决定把 AI 工作流从「本地控制远端」**演进**为「远端就是 AI 工作空间」——这是 ch02 终局方向之一。

### 2. 方案 C2：双 MCP 拆分部署（本地嵌入式 + 远端文件）

C1 的一个痛点是：把现有 MCP server 整体搬到远端后，PowerShell 工具、serial 工具、port_scan 等面向本地物理设备的工具就失去意义，要做裁剪（见 C1 缺点）。C2 的思路是**不搬整体，只拆文件能力**——本地 MCP 不动，另起一个轻量远端 MCP 专门管文件读写和搜索。

#### 2.1 做法

- 本地嵌入式 MCP **保持原样不动**——继续管 ssh、serial、adb、powershell、port_scan 等设备交互工具；
- 在编译服务器上**新建一个独立的轻量 MCP server**（Linux 版），只注册文件相关工具（`remote_file_read` / `remote_file_write` / `remote_file_edit` / `remote_grep` / `remote_glob` / `remote_build`），以 stdio 方式运行；
- 本地 MCP client 配置里注册两个 server，第二个通过 SSH exec 接入远端（同 C1，不开远程端口）。

```
+---------- Windows -----------+                       +------- 编译服务器 ------+
| claudecode/opencode          |                       |                       |
|  ├── 嵌入式 MCP (本地)         | stdin/stdout          |  (不部署)              |
|  │   ssh / serial / adb /    |                       |                       |
|  │   powershell / port_scan  |                       |                       |
|  │                           |                       |                       |
|  └── 文件 MCP (远端)          | MCP over SSH/JSON-RPC | 文件 MCP server (Linux)|
|      remote_file_read/write  | <==================>  | file tools (本地 fs)   |
|      remote_grep / glob      |                       | rg / cscope (本地)     |
|      remote_build            |                       | make / git (本地)      |
+------------------------------+                       +-----------------------+
```

MCP client 配置示例：

```json
{
  "mcpServers": {
    "embedded": {
      "command": "node",
      "args": ["./bin/embedded-mcp-toolkit-cli.js"]
    },
    "remote-src": {
      "command": "ssh",
      "args": ["user@build-server", "node /opt/mcp-file-server/cli.js"]
    }
  }
}
```

MCP client 会把两个 server 的工具合并成一份列表给 AI，AI 看到的是统一的工具池，不关心它们来自哪个进程。

#### 2.2 优缺点

| 优势 | 局限 |
|------|------|
| **保留 C1 的文件操作优势**——远端文件 MCP 的 handler 就是 `fs.readFile` / `execFile("rg")` 等纯本地调用，零网络 I/O，搜索极快 | **两个 MCP 进程**——AI 工具列表里多了一批前缀不同的工具，需要靠 description 引导 AI 选对（如读源码用 `remote_file_read` 而非 `ssh_shell_exec "cat"`） |
| **本地嵌入式 MCP 零改动**——serial / adb / powershell 这些绑定本地物理设备的工具原样保留，不用裁剪、不用跨平台移植 | **工具命名不能撞**——两个 server 的工具名需区分前缀（如 `remote_` / `src_`），避免重名冲突 |
| **远端 MCP 极轻**——只注册文件工具，不依赖 ssh2 / serialport / adb 等库，部署到 Linux 几乎无障碍 | **远端仍需 Node.js**——虽然远端 MCP 很轻，但仍要求编译服务器有 Node.js 运行时 |
| **职责清晰**——设备交互走本地 MCP，源码操作走远端 MCP，各司其职，不存在状态同步或协调问题 | **通道不统一**——与 C1 的「单一远程执行环境」相比，文件操作和设备操作走两个进程、两个通道，架构统一性不如 C1 |
| **渐进式落地**——可以先跑通本地嵌入式 MCP，再按需加远端文件 MCP，不需要一步到位 | - |

#### 2.3 读写机制

本节对照 B2（2.3）说明 C2 在读写上的根本优势。

##### 2.3.1 发生在哪一端

**文件 MCP server 跑在编译服务器远端，读写全部在远端本地完成。** 与 B2 相反——B2 的 MCP server 跑在 Windows 本地、数据在远端、每次读写都要过网；C2 的 MCP server 就跑在数据旁边，文件工具的 handler 是 `fs.readFile` / `execFile("rg")` 等纯本地调用，零网络 I/O。

```
B2 read：ssh2.sftp.readFile() → SFTP 协议层 → SSH 隧道 → 远端 sftp-server → 读磁盘（~30ms/次）
C2 read：fs.readFile()        → 内核直接读磁盘（~1ms/次）

B2 grep：ssh exec "rg ..." → SSH 通道建立 + 执行 + 回传（~50-100ms/次）
C2 grep：execFile("rg")    → 本地 fork + 直接拿结果（~10-20ms/次）
```

**支持直接写回远端**。AI 调用远端文件 MCP 的写工具修改文件，修改直接发生在编译服务器本地，与 C1 写回能力相同。

##### 2.3.2 既省 token 又省网络（对照 B2 的「假局部」）

C2 的工具 API 可以和 B2 设计得一模一样——都支持 `remote_file_read(path, offset, limit)` 按行读、`remote_file_edit(path, old_str, new_str)` 局部 patch。对 AI 来说 token 消耗相同，但**底层网络传输量完全不同**：

| 操作 | AI 视角（省 token） | C2 底层网络 |
|------|-------------------|------------|
| 局部读 20 行 | 一样，只收 20 行 | **零网络**（本地 fs 读） |
| patch 改 1 行 | 一样，只发 old_str/new_str | **零网络**（本地 fs 读+写，只把结果回传） |
| 全量读 | 一样 | 零网络 |
| 全量写 | 一样 | 零网络 |

对比 B2 的「假局部」（2.3.2）：AI 改 10 个文件，B2 底层传了 20 次完整文件内容过网，C2 全是本地操作。这是 C2 性能优势的一个隐藏放大因素。

##### 2.3.3 内存还是磁盘

**替换在远端进程内存完成，直接落远端磁盘。** 与 B2 一样是「读全文 → 内存替换 → 全量覆盖写回」，但因为全程在远端本地，不涉及网络搬运。

##### 2.3.4 是否全量覆盖：写底层仍是全量覆盖，但本地 fs 安全等级高

**（1）同样是全量覆盖写，本地和远端网络的安全性天差地别**

不管是 C2 还是 B2，底层写文件都是同一个流程（读全文 → 内存替换 → `O_TRUNC` 清空 + 全量写回）。可怕的是「清空后写到一半被打断」，两种方案的中断风险完全不同：

```
C2（本地 fs.writeFile）：
  open(O_TRUNC)   ← 清空文件，~微秒级
  write(data)     ← 写入，~1-2ms
  close()
  整个窗口：~1-2ms

B2（SFTP writeFile）：
  SFTP OPEN(O_TRUNC)    ← 远端文件被清空，~30ms（网络往返）
  SFTP WRITE chunk 1    ← 传第一个块，~30ms
  ...
  SFTP CLOSE            ← ~30ms
  整个窗口：~60ms 到数秒（取决于文件大小）
```

关键区别不是窗口大小本身，而是**中断的原因和概率**：

| 中断类型 | C2（本地） | B2（网络） |
|---------|-----------|-----------|
| 断电 | 极罕见，文件系统有日志保护 | — |
| 进程崩溃 | 罕见 | — |
| 网络抖动 | — | **常见**（WiFi 波动、VPN 掉线） |
| SSH 会话超时 | — | **常见**（空闲超时、服务器 kill） |
| 服务器重启 | — | 偶发 |

本地操作的中断是「极端事件」，网络操作的中断是「日常事件」——概率差了几个数量级。

**（2）两种方案的实际安全等级**

| 写法 | C2 | B2 |
|------|----|----|
| 直接覆盖写（简单写法） | `fs.writeFile` 清空后写，窗口 ~1ms，中断是极端事件，**实践中安全** | `sftp.writeFile` 清空后写，窗口 ~60ms+，网络断开有**文件损坏风险** |
| 原子写（安全写法） | `write tmp + rename`，全本地，原子窗口微秒级 | `sftp.write tmp + sftp.rename`，每步过网络，但原文件不会被损坏 |
| ssh2 库默认行为 | — | `sftp.writeFile()` **直接覆盖原文件**，不走临时文件 + rename |

关键差异：

- C2 **即使不写临时文件**，直接 `fs.writeFile` 覆盖写，因为窗口只有 1-2ms，实际上也不会出问题；
- B2 **必须写临时文件 + rename**才安全，否则网络断开时原文件可能被清空到一半。

**（3）无模糊失败**

C2 的 `fs.writeFile` 在本地进程内完成，成功/失败立刻知道，非此即彼。而 B2 的写入确认要过网络回来，如果回复丢失，MCP 不知道文件到底写了没有（见 B2 的 2.3.5 模糊失败）。C2 没有这个隐患。

#### 2.4 适用场景

- 想享受 C1 的文件操作性能，但不想整体迁移现有 MCP server 的场景；
- 嵌入式设备交互（serial / adb / powershell）是刚需、且必须跑在本地 Windows 上的场景；
- 渐进式落地——先跑本地嵌入式 MCP，再加远端文件 MCP。

### 3. 方案 C3：镜像部署（agent 在远端 + 本地设备 MCP）

C2 的拓扑是「agent 在 Windows + 文件 MCP 在编译服务器」。一个自然的追问是：**既然文件 MCP 能跑在远端、本地 agent 通过 SSH stdio 接入，那么反过来行不行——agent 跑在编译服务器，嵌入式 MCP 跑在 Windows，照样走 SSH stdio？**

答案是：**协议层面完全对称，工程层面连接方向反转了**。本方案记录这个镜像拓扑。

#### 3.1 做法

- claudecode / opencode / zcode 直接跑到**编译服务器**上，以 stdio 方式运行；
- 嵌入式 MCP server **保持在 Windows 本地**不动——继续管 serial、adb、powershell、port_scan 等设备交互工具；
- 编译服务器上的 agent 通过 SSH exec 反向接入 Windows 本地的 MCP（agent 当 MCP client，Windows 当 MCP server）。

```
+---------- Windows -----------+                         +------- 编译服务器 ------+
| 嵌入式 MCP (本地)              |                         | claudecode/opencode    |
|  ssh / serial / adb /        |  MCP over SSH/JSON-RPC  |  └── 文件操作 (内置)    |
|  powershell / port_scan      | <==================>    |      read / edit / write|
|                              |   反向 SSH 通道          |      (本地 fs, 零网络)  |
|  (本地 USB / 串口 / 烧录器)     |                        |      grep / glob (本地) |
+------------------------------+                        |      make / git (本地)  |
                                                     +-----------------------+
```

注意 C3 与 C2 的一个关键差异：**文件操作不需要 MCP**。agent 本身就在源码旁边，文件就在它的本地文件系统里，所以 AI 直接用 agent 内置的 read / edit / write / grep 工具——和本地开发完全一样。只有设备交互才需要走 MCP over SSH 反向通道。

编译服务器上的 agent MCP client 配置示例（配置文件在编译服务器上，如 `~/.config/<agent>/mcp.json`）：

```json
{
  "mcpServers": {
    "embedded": {
      "command": "ssh",
      "args": ["user@windows-host", "node C:\\path\\to\\embedded-mcp.js"]
    }
  }
}
```

对比上文 C2 的配置（第六节 2.1）：C2 要注册两个 server（本地嵌入式 + 远端文件），C3 只需注册一个反向 MCP——因为文件操作由 agent 内置工具接管，不占 MCP 名额。配置更简单，是 C3 的一个隐性优势。

#### 3.2 关键障碍：SSH 连接方向反转

MCP 的 stdio 传输就是「stdin/stdout 管道」，SSH 把远端进程的 stdio 接到本地——这个机制不关心谁在远端、谁在本地，**协议层完全对称**。但「SSH 连接谁主动连谁」不是对称的，C3 撞到三堵墙：

| 障碍 | 说明 |
|------|------|
| **Windows 要开 SSH server** | OpenSSH for Windows 可装，但默认未启用；工位机通常不当作 SSH 服务端 |
| **网络可达性** | 编译服务器常在机房/内网，Windows 工位机在办公网/防火墙后——「服务器主动连工位机」经常不通 |
| **需要反向隧道或 VPN** | 要么 Windows 工位机主动建反向 SSH tunnel（`ssh -R`），要么两端在同一个 VPN 里 |

这三条都不是「MCP 协议问题」，是**网络拓扑问题**。C2 之所以顺，是因为它顺着 SSH 的自然方向走（工位机是 SSH 客户端、服务器是 SSH 服务端）；C3 是逆着走（服务器要当 SSH 客户端、工位机要当 SSH 服务端）。

#### 3.3 优缺点

| 优势 | 局限 |
|------|------|
| **文件操作零网络**——agent 和源码同机，read/edit/write/grep 全是本地 fs 调用，性能等同本地开发，是所有方案里文件性能最好的 | **SSH 方向反转**——Windows 要开 SSH server，网络可达性是首要障碍（见 3.2） |
| **配置最简**——只需注册一个反向 MCP，文件操作由 agent 内置工具接管，不像 C2 要配两个 server | **交互界面损耗**——agent 是 TUI 程序，跑在编译服务器上，用户要 `ssh` 进远端终端操作，TUI 渲染经 SSH 传回，刷新/颜色/键盘事件可能有损耗 |
| 本地嵌入式 MCP 零改动，设备工具原样保留 | **设备操作过网**——每次 serial/adb/powershell 调用都要走反向 SSH 通道，延迟高于 C2 的本地直调 |
| 与 ch02「跨边界设备访问」方向一致——agent 在远端能反过来操作本地物理设备 | 反向通道稳定性依赖网络，VPN/反向隧道断开则设备工具全部失联 |

#### 3.4 读写机制

C3 是 C2 的镜像——把「文件过网、设备零网络」翻转成了「文件零网络、设备过网」。两端的读写机制分别说明。

##### 3.4.1 文件读写：零网络，agent 内置工具

**agent 就跑在源码旁边，文件操作全部是远端本地调用。** AI 用的是 agent 内置的 read / edit / write / grep 工具（不是 MCP 工具），因为文件真的在 agent 的本地文件系统里——这点与 A1 / B3 相同（见第二节「AI 的文件读写工具」的两类来源）。edit 的字符串替换发生在远端 agent 进程内存，底层全量覆盖写落远端磁盘，和普通本地开发完全一样，写中断风险极低（同 C2 的文件写安全等级）。

##### 3.4.2 设备交互：过反向 SSH 通道

**嵌入式 MCP 跑在 Windows 本地，但 agent 在远端，每次设备操作都要过反向 SSH 通道。** AI 调用 `serial_write` / `adb_shell` / `powershell_exec` 等工具时，请求经反向 SSH 通道到达 Windows 本地的 MCP server，MCP 操作本地物理设备后把结果回传。

这和 B2 的「过网」模式形态相似，但有两点不同：

- **方向相反**——B2 是本地 agent → 远端文件，C3 是远端 agent → 本地设备；
- **无覆盖写风险**——设备操作通常是「发命令 + 等回显」，失败可重试，不像文件覆盖写那样有「清空后中断导致损坏」的风险。反向通道断开的最坏结果是设备操作失败，不会损坏设备状态。

##### 3.4.3 本质：一次代价交换

把 C2 和 C3 并排放，能看清这次交换的全貌：

| | C2（agent 在本地） | C3（agent 在远端） |
|---|---|---|
| AI agent 位置 | Windows 本地 | 编译服务器远端 |
| 文件读写 | **过网**（SSH 到远端文件 MCP） | **零网络**（agent 内置工具，同机） |
| 设备交互（serial/adb/USB） | **零网络**（agent 与 MCP 同机） | **过网**（反向 SSH 到本地 MCP） |
| SSH 方向 | 正向（自然） | 反向（需额外配置） |

所以这不是「谁更好」的单向优劣，而是一次代价交换：

- 如果工作流是**重度改代码 + 偶发烧录调试** → C3 更优，文件零网络是主战场，设备反向连接频率低；
- 如果工作流是**频繁烧录/串口交互 + 代码相对稳定** → C2 更优，设备零网络，文件过网可接受。

嵌入式开发的典型循环是「改代码 → 编译 → 烧录 → 调试」，其中 grep/read 频率远高于烧录，**单纯从操作频率看 C3 占优**。但要扣掉反向 SSH 的工程复杂度——如果为了省文件网络而引入反向隧道的运维负担，未必划算。

#### 3.5 适用场景

- 编译服务器与 Windows 工位机网络互通良好（同 VPN / 反向隧道已就绪），反向 SSH 不是障碍；
- 工作流以**源码分析与修改为主**，设备烧录/调试为辅——文件零网络是主要收益；
- 团队能接受 `ssh` 进编译服务器操作 agent 的交互模式（TUI over SSH）。

### 4. B2 与 C2 按开发阶段的性能对比

C2 和 B2 表面都是「MCP 在本地 + 操作远端文件」，但两者的性能差距随工作模式不同而显著变化。关键在于 **grep 和 read 在哪跑**（C2 在远端本地、B2 过网）。

#### 4.1 阶段一：前期了解项目（陌生项目架构分析）

此阶段是 **read + grep 的高频放大场景**——AI 要理解陌生项目，会疯狂 grep 找入口、找定义、找调用链，然后批量读文件理解架构。

| 维度 | 前期了解项目 |
|------|------------|
| 操作类型 | 几乎纯读 + 大量搜索 |
| 读文件量 | 大（几十到上百个，广撒网） |
| grep 频率 | 极高（换关键词、换目录反复搜） |
| 写操作 | 几乎没有 |
| 核心瓶颈 | **搜索 + 批量读** |

典型一次探索：跑 30 次 grep + 读 80 个文件。

```
B2：30 × 80ms(grep) + 80 × 30ms(read) ≈ 2.4s + 2.4s = ~5s 纯网络等待
C2：30 × 15ms(grep) + 80 × 1ms(read)  ≈ 0.5s + 0.08s = ~0.6s
```

**前期了解项目阶段，C2 优势最大**——网络 I/O 累积严重，B2 的逐文件过网会明显拖慢 AI 的探索节奏。

#### 4.2 阶段二：后期开发读写

进入开发阶段后，工作模式变成定向操作——AI 已了解架构，操作聚焦在少数文件。

| 维度 | 后期开发读写 |
|------|------------|
| 操作类型 | 定向读 + 频繁写 + 编译 |
| 读文件量 | 小（定向看几个文件） |
| grep 频率 | 低（偶尔确认接口/定义） |
| 写操作 | 频繁（edit、write） |
| 核心瓶颈 | **写回 + 编译** |

每次只读 1-2 个文件，B2 的 SFTP 单文件读 30ms 完全可接受；写操作两者都是直接写回远端，差异在几十毫秒级，不影响体验。**后期开发阶段，两者差距缩小，但 C2 仍优**——尤其在编译环节，C2 的 `execFile("make")` 流式输出比 B2 的 `ssh exec "make"` 回传更稳定。

#### 4.3 结论

```
前期了解项目 → C2 明显更好（搜索 + 批量读是主战场，网络 I/O 累积严重）
后期开发读写 → C2 仍更好，但差距不大（定向操作为主，B2 也够用）
```

**两个阶段 C2 都更优，但前期差距远大于后期。** 如果正在评估一个陌生项目，正是 C2 优势最大的场景。

### 5. 方案选择建议

| 你的诉求 | 推荐方案 |
|---------|---------|
| **最快验证、零开发** | B3 sshfs / SMB 挂载（但只限小项目/临时用；SMB 体验优于 sshfs，搜索仍是瓶颈） |
| **稳健落地、开发量适中** | B2 + SFTP |
| **设备交互是刚需 + 想享受 C1 文件性能，但不想整体迁移** | C2 双 MCP 拆分部署 |
| **统一架构、面向未来、完整闭环** | C1 JSON-RPC 远程服务进程 |
| **源码分析为主 + 反向 SSH 已就绪，想极致文件性能** | C3 镜像部署 |

C1 和 C2 的选择关键在于**是否需要保留本地嵌入式设备工具**：

- 如果嵌入式设备交互（serial / adb / powershell）是刚需且必须跑在本地 Windows 上 → **选 C2**，本地 MCP 不动 + 远端加轻量文件 MCP，各司其职；
- 如果目标是把 AI 工作流整体演进为「远端就是 AI 工作空间」，可以裁剪掉本地设备工具 → **选 C1**，单一远程执行环境，架构最统一。

C2 是 C1 的渐进版——保留了 C1 的文件操作性能优势，同时避开了 C1 的整体迁移和裁剪痛点，适合「设备交互 + 源码操作」并重的嵌入式开发场景。

#### 5.1 C2 vs C3：一对镜像，代价交换

C3 是 C2 的镜像——拓扑对称，但代价交换（见 C3 的 3.4.3）：

| | C2（agent 在本地） | C3（agent 在远端） |
|---|---|---|
| 文件读写 | 过网（SSH 到远端文件 MCP） | **零网络**（agent 内置工具，同机） |
| 设备交互 | **零网络**（agent 与 MCP 同机） | 过网（反向 SSH 到本地 MCP） |
| SSH 方向 | 正向（自然） | 反向（需额外配置） |

选 C3 的前提是**反向 SSH 能打通且网络稳定**——如果编译服务器与 Windows 工位机网络互通良好（同 VPN / 反向隧道已就绪），且工作流以源码分析为主、设备交互为辅，C3 的文件性能是所有方案里最好的。反之，如果反向通道搭建困难或不稳定，C2 的正向连接更省心。

#### 5.2 C2 vs B2：按开发阶段选择

两者的关系不是「二选一」，而是**递进**：

```
B2：低门槛起步——服务器不方便装 Node，或想快速验证
 ↓ 服务器有 Node.js，且想要更好的性能和统一通道
C2：升级——远端常驻 MCP，文件操作零网络，搜索也更快
```

如果服务器确定能装 Node.js，B2 可被 C2 取代；如果不能，B2 不可替代。

| 场景 | B2 | C2 | 谁赢 |
|------|----|----|------|
| 服务器能装 Node.js | 可用 | 可用 | **C2 赢**——read 走本地 fs，grep 走本地 exec，全零网络，通道统一 |
| 服务器不能装 Node.js | 可用 | 不可用 | **B2 赢**——C2 直接不可行 |

## 八、 结论

- **只读分析**：B1（远端索引查询）是有效辅助，但不能独立完成改代码闭环；文件摘要应作为 B2 / C1 / C3 工具层里的一个高级 tool 实现，不是独立方案。
- **短期最实用**：A1（rsync 拉本地）+ B1（远端索引辅助），零开发，30 分钟可用；缺点是改完代码后需要额外同步步骤。
- **中期最值得做**：B2（MCP server 增设文件工具层，优先 SFTP 实现）——AI 直接读写远端文件，无本地副本，无同步步骤，是 ch02 主力方向之一。注意读写粒度：`ssh_file_read` 走局部读；写回需同时提供 `ssh_file_write`（全量覆盖）和 `ssh_file_edit`（局部 patch + baseline 校验）两种粒度，后者是 AI 改大文件的主力工具。
- **五种可落地方案**：C1 JSON-RPC 远程服务进程（统一、可扩展）、C2 双 MCP 拆分部署（设备交互 + 文件操作各管各的）、C3 镜像部署（agent 在远端 + 反向 SSH 接本地设备 MCP）、B2 + SFTP（稳健折中）、B3 sshfs / SMB 挂载（最小开发量但稳定性差，搜索性能是硬伤——SMB 也只能缓解单次 I/O，无法根治批量搜索问题，见 3.2.2）。
- **镜像拓扑 C3**：MCP 协议层面对称，真正的障碍在网络拓扑——反向 SSH 是工程核心难点，不是协议问题。本质是一次代价交换：用「文件零网络」换「设备零网络 + 反向连接复杂度」。源码分析为主、反向 SSH 能打通时，C3 的文件性能是所有方案里最好的。
- **嵌入式场景推荐**：C2 双 MCP 拆分部署——本地嵌入式 MCP 管 serial / adb / powershell，远端轻量 MCP 管文件读写和搜索，兼顾设备交互刚需和 C1 级文件操作性能，且本地 MCP 零改动。
- **远期终局**：C1（远端跑 MCP server，本地 AI 远程接入），与 ch02「跨边界设备访问」终极目标对齐。

落地节奏建议：**先把 A1 + B1 跑通验证 AI 分析质量，再投入 B2 或 C2 实现远程直接读写**——避免过早抽象。C2 尤其适合嵌入式场景：本地嵌入式 MCP 已就绪，只需在远端新建一个轻量文件 MCP 即可补齐源码操作能力。若反向 SSH 能稳定打通且工作流以源码分析为主，可进一步评估 C3 镜像部署——文件性能最佳，但设备操作需过反向通道。
