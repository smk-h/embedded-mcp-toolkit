# 配置文件分文件化 Spec

## 背景

`embedded-mcp-toolkit` 的所有设备配置集中在单个 `.embedded/configs/config.yaml` 中。实测统计：`keyProvider` 段在该文件中重复出现 14 次，文件已有 6~7 个设备、191 行，且设备数会继续增长。

当前文件布局带来三个问题：

- **单文件过长**：新增设备时文件持续膨胀，定位/修改某个特定设备需要在长文件中滚动翻找。
- **改动耦合**：修改某台设备时整个大文件都处于编辑状态，Git 层面产生不必要的 diff 噪声，多人维护易冲突。
- **设备画像不自包含**：了解一台设备的完整配置需要在同一文件内拼凑相距较远的多个片段。

关键前提：各设备的配置参数**本质上并不相同**（host / port / 用户名 / 密码 / 串口号 / keyProvider 各异）。当前文件里多台设备配置看起来相同，仅是测试期间为方便临时写的一样。因此问题本质是**组织方式不便于定位和增长**，而非"可继承的重复太多"。

## 目标

- 将单文件 `config.yaml` 中的设备配置拆分为 `devices/` 目录下的独立文件，每台设备一个文件、自包含。
- 加载层同时兼容「旧单文件布局」和「新分文件布局」，现有用户无需被迫迁移即可升级。
- 提供一键迁移命令，帮助现有用户从旧布局切换到新布局。
- 保持所有对外行为（MCP 工具调用、`DEVICE` 环境变量、设备名解析、设备列表）不变。

## 功能需求

- **F1：分文件布局支持**。主配置 `config.yaml` 只承载 `default`（默认设备名）等全局设置；每台设备的完整配置（ssh / serial / adb / 各自 keyProvider）存放在 `devices/<设备名>.yaml`，一个文件一台设备。

- **F2：设备目录自动发现**。在分文件布局下，加载层自动扫描 `devices/` 目录，将其下每个 `.yaml` 文件的文件名（去掉扩展名）作为设备名加载，用户无需在主配置中显式维护设备名单。

- **F3：向后兼容旧单文件布局**。当 `devices/` 目录不存在或为空时，加载层回退到从单个 `config.yaml` 的 `devices` 段读取设备配置，与重构前行为完全一致。两种布局由加载层自动识别，对调用方透明。

- **F4：布局自动判定**。加载层依据 `devices/` 目录是否存在且非空，自动判定使用分文件还是单文件布局，无需用户额外配置开关。两种布局不混用：一旦识别到有效设备文件，以分文件布局为准。

- **F5：配置拆分命令**。提供 CLI 命令读取旧单文件 `config.yaml` 的 `devices` 段，将每个设备导出为独立的 `devices/<设备名>.yaml` 文件。命令具备覆盖保护（目标已存在时默认不覆盖，可强制覆盖），并清晰报告处理结果（创建了哪些文件、跳过了哪些）。

- **F6：init 命令适配新布局**。`init` 命令生成的初始配置采用分文件布局：主 `config.yaml` 加一个 `devices/board-<默认设备名>.yaml` 示例设备文件，引导新用户使用新布局。

- **F7：对外行为不变**。`resolveDeviceName()`、`listDevices()`、`getSSHConfig()`、`getSerialConfig()`、`getAdbConfig()`、`getKeyProviderConfig()`、`getAllConfig()` 等对外函数的语义与返回结构保持不变；`DEVICE` 环境变量、`BOARD_CONFIG_PATH` 环境变量的作用与优先级保持不变。

## 非功能需求

- **N1：加载性能不退化**。分文件布局引入多次文件读取，但因配置层存在进程级单例缓存（启动时加载一次后整会话复用），单次加载的亚毫秒级增量在 MCP server 整个生命周期中可忽略。重构后单次加载耗时应保持在毫秒级以内。

- **N2：向后兼容为零感知**。现有用户升级后，旧的单文件 `config.yaml` 应能被原样读取，设备配置无丢失，无需用户做任何改动。

- **N3：不引入新依赖**。本次改动仅使用项目已有的 `js-yaml`、Node `fs` 模块，不新增第三方依赖。

- **N4：代码风格一致**。与 `src/shared/config.ts` 现有风格保持一致（相对 import 带 `.js` 后缀、JSDoc 注释风格、命名约定）。

- **N5：Git 历史连续**。对现有文件的修改通过常规编辑保留历史；`split` 命令生成的设备文件为新建文件。

## 不做的事

- **不引入默认值继承机制**（方案 B）：不增加顶层 `defaults` 段，不做深度合并。原因：各设备参数本质不同，可继承的公共默认值极少，投入产出比低。本次只做分文件化。
- **不引入 YAML 锚点机制**（方案 A）：不使用 `&kp` / `*kp` 引用语法，避免给用户增加 YAML 高级语法的学习成本。
- **不改配置文件格式**：仍为 YAML，不转换为 JSON。
- **不改 MCP 工具行为**：工具注册、工具名称、工具入参/出参结构均不变。
- **不修改密码等敏感信息的存储方式**：设备文件仍明文存放（与现状一致），`.gitignore` 策略仅针对新增的设备文件做对应扩展。

## 验收标准

- **AC1（F1+F2）**：在 `devices/` 目录下放置若干 `<设备名>.yaml` 文件，调用 `listDevices()` 返回的设备名集合与目录下的文件名（去扩展名）一致。

- **AC2（F3）**：删除或清空 `devices/` 目录后，`listDevices()` 回退为读取单文件 `config.yaml` 的 `devices` 段，返回结果与重构前一致。

- **AC3（F4）**：同时存在 `devices/` 目录（含设备文件）和单文件 `config.yaml`（含 `devices` 段）时，加载层以分文件布局为准，单文件中的 `devices` 段被忽略；主 `config.yaml` 的 `default` 字段仍生效。

- **AC4（F5）**：执行配置拆分命令后，旧 `config.yaml` 中 `devices` 段的每个设备均生成对应的 `devices/<设备名>.yaml`，文件内容包含该设备的完整配置（ssh/serial/adb/keyProvider 无丢失）。

- **AC5（F5 覆盖保护）**：目标设备文件已存在时，拆分命令默认跳过该文件并报告跳过，加 `--force` 后才覆盖。

- **AC6（F6）**：在空目录执行 `init` 后，生成 `.embedded/configs/config.yaml`（含 `default`）和 `.embedded/configs/devices/board-<默认设备名>.yaml`（示例设备文件）。

- **AC7（F7）**：对同一台设备（如 `board-a`），无论配置来自分文件还是单文件，`getSSHConfig("board-a")` / `getSerialConfig("board-a")` / `getAdbConfig("board-a")` 返回的内容一致。

- **AC8（F7）**：设置 `DEVICE=board-b` 后，`resolveDeviceName()` 在两种布局下均返回 `"board-b"`。

- **AC9（N1）**：MCP server 从分文件布局启动时，启动日志输出 `Config loaded` 且无性能告警，工具调用可正常解析设备配置。

- **AC10（N3）**：`git diff package.json` 无新增依赖条目。
