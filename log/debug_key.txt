================================================================================
  流程一: 【错误】 shell_send "debug" (单独发送)
================================================================================

根因:
  shell_send 在命令后自动追加 echo "__END_MARKER_xxx__" 结束标记。
  实际发送到 PTY: "debug\n" + "echo __END_MARKER__\n"
  echo 标记在 PSH 进入 key> 等待时被当作密钥读入 → Invalid key!

--------- 终端原始输出 ---------
locked> debug
echo "__END_MARKER_1779532917247__"

=== DEBUG MODE ===
Challenge Code: PSH-6DBB-E36C-06F9-390A

Enter key to unlock: key>
[PSH] Invalid key! Returning to locked mode.
locked>
--------- 终端输出 END ---------


================================================================================
  流程二: 【正确】 shell_send "debug\n123456" (合并发送)
================================================================================

解决:
  debug 和密钥 123456 合并发送，实际发送:
    "debug\n" + "123456\n" + "echo __END_MARKER__\n"
  123456 在 key> 提示出现时恰好位于输入缓冲区头部，被正确读取。

--------- 终端原始输出 ---------
locked> debug
123456
echo "__END_MARKER_1779532925478__"

=== DEBUG MODE ===
Challenge Code: PSH-86BF-7F04-E5B0-E040

Enter key to unlock: key>
[PSH] Access Granted! Unlocking shell...

root@ATK-IMX6U:~# echo "__END_MARKER_1779532925478__"
__END_MARKER_1779532925478__
root@ATK-IMX6U:~#
--------- 终端输出 END ---------


================================================================================
  诊断结论
================================================================================
1. shell_send 的 echo 标记与 PSH interactive prompt (key>) 存在时序冲突
2. echo 标记在 debug 切换后抢占 key> 输入窗口，阻塞用户密钥输入
3. Workaround: 将 debug 与密钥合并到一次 shell_send 调用
4. 根本解决: shell_send 需支持 interactive prompt 感知，或为
   shell_unlock 配置正确的 unlock sequence
================================================================================
