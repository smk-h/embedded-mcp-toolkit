## 一、 背景

当前 adb、ssh、serial 三个通道的交互式 shell exec 工具(adb_shell_exec / ssh_shell_exec / serial_exec)采用「write 命令 → sleep 固定时长 → read 缓冲区」的盲式三步模式,没有任何「命令是否结束」的判断机制。

该模式对 `ls`、`getprop` 这类瞬时返回的命令工作正常,但对 `logcat`、`ping`、`top` 这类前台常驻命令会彻底失效:

- 工具拿到一段输出后即返回,但命令本身未退出,仍在后台持续往缓冲区灌数据
- 下一次 exec 的输出与残留的常驻命令输出混在一起,导致输出错乱
- 调用方(含 LLM)试图发送 Ctrl+C / Ctrl+U 终止命令时,因当前工具体系没有正确发送控制字符的入口(write 默认补 `\n` 破坏控制字符语义),并非每次都能成功

此外,三个通道的 exec 实现是复制粘贴的,存在代码重复。

## 二、 目标

- 让 `logcat`/`top`/`ping` 等常驻命令执行后能被可靠终止,不再污染后续操作
- 让调用方能确定性地发送 Ctrl+C / Ctrl+U / Ctrl+D / Ctrl+Z 等控制字符,不再靠 write 碰运气
- 统一三个通道的 exec 行为,消除重复实现
- 不影响现有瞬时命令(`ls`/`getprop` 等)的正常使用,并尽量让它们更快返回

## 三、 功能需求

### F1:发送控制字符工具(send_ctrl)

为每个通道(adb/ssh/serial)提供独立的发送控制字符工具,接收会话 ID 和控制字符类型,以不追加换行的方式写入,保证控制字符语义正确。支持 Ctrl+C(`\x03`)、Ctrl+U(`\x15`)、Ctrl+D(`\x04`)、Ctrl+Z(`\x1a`)。

### F2:exec 前置冲刷

每个通道的 exec 在发送命令前,先清空并丢弃输出缓冲区中的残留数据,避免上次未终止命令的输出混入本次结果。

### F3:exec 超时熔断

为每个通道的 exec 增加最大执行时长参数(默认 10 秒,可由调用方覆盖)。在该时长内轮询输出缓冲区:

- 检测到 shell 提示符 → 判定命令已结束,立即返回已有输出,不必等满时长
- 超过最大执行时长仍未检测到提示符 → 自动发送 Ctrl+C 终止命令,返回已有输出并标注「命令被中断」

### F4:exec 提示符检测

实现 shell 提示符识别,用于判定命令是否结束。支持常见的 Android(`:/ $`、`:/ #`)、Linux(`$`、`#`、`>`)等提示符,且允许通过设备配置覆盖提示符正则。

### F5:exec 复用统一逻辑

将三个通道的 exec 主体逻辑(前置冲刷、轮询、提示符检测、超时熔断)抽到共享层,各通道只保留「写命令」「读缓冲区」「发控制字符」的传输层差异,消除复制粘贴。

### F6:adb PTY 分配与回显剥离

adb shell 默认不分配 PTY,设备侧 shell 不回显提示符,导致 F4 提示符检测在 adb 通道失效。需给 adb shell 加 `-t -t` 强制分配 PTY,使其回显提示符。PTY 模式会同时回显输入的命令行(如 `:/ $ echo hi`),exec 需剥离首行回显,只保留真实输出。

## 四、 非功能需求

### N1:向后兼容

现有的瞬时命令(`ls`/`getprop` 等)行为不变,输出内容与现在一致;现有的 write/read/close 工具签名和行为完全不动。exec 的原有参数(session_id、command、delay、clear)全部保留,新增的 maxDuration 为可选参数有默认值,不破坏既有调用。

### N2:传输层改动最小化

复用现有 BaseShell 的 write/read 能力,提示符检测和熔断逻辑在工具层处理。**例外**:`transports/adb.ts` 需把 `shell` 改为 `shell -t -t` 强制分配 PTY(真机验证发现:adb 默认无 PTY 不回显提示符,提示符检测在 adb 通道完全失效)。ssh 通道本就有 PTY,serial 通道走物理串口,两者不改。改动仅限 spawn 参数,不涉及连接/读写逻辑。

### N3:性能

对瞬时命令,新版 exec 因提示符检测能比现在更快返回(不必傻等满 delay);轮询间隔应合理(不占用过多 CPU,也不让延迟过大)。

### N4:可观测性

控制字符发送、熔断触发、前置冲刷等关键事件都记录到现有日志体系(logger.info/error),便于排查「为什么命令被中断了」「为什么提前返回了」。

### N5:确定性

发送控制字符工具必须保证不追加任何换行符或额外字节,发出的字节与调用方指定的一致;这是当前 write 工具最大的不可靠来源,新工具必须从机制上杜绝。

### N6:跨通道一致性

三个通道的 send_ctrl、熔断行为、提示符检测规则保持一致,调用方无需为不同通道学习不同用法。

## 五、 不做的事

### 1. 不实现方案 C(有限采样工具)

user.md 中提到的 `*_shell_capture(command, duration_ms)` 专用采样工具本次不做。A+B 组合已能覆盖 `logcat`/`top`/`ping` 的核心痛点(自动熔断 + 可靠停止);capture 作为未来增强项,等 logcat 场景需要更精确的「采样 N 秒」语义时再引入。

### 2. 只改交互式 `*_shell_exec`,不动其他命令执行路径

各通道的非交互式命令执行路径本次都不改,原因如下:

- **`adb_exec`**:通过 spawnSync 独立子进程执行,每次调用进程隔离,**不会产生本次要解决的「buffer 污染下一次调用」问题**(子进程输出不共享)。需注意它有自己的局限——spawnSync 同步阻塞会卡住整个 MCP server 直到超时,且超时后丢弃已收集的输出,因此它本就只适合一次性短命令(devices/install/push 等),不适合跑 logcat 这类常驻命令。其同步阻塞与超时丢输出属于另一问题域,本次不改,可另立项优化
- **`ssh_build`**:虽然复用交互式 shell 会话执行命令,但内部已自带完成标记(`___MCP_BUILD_DONE___:$?`)检测、`shell.drain()` 残留排空、`maxWait` 超时轮询机制,等价于本次方案 B 的能力,无需重复改造
- **ssh / serial 的一次性命令**:这两个通道没有独立的 spawnSync 非交互式工具,它们的一次性命令执行就是本次要改造的 `*_shell_exec`,已在覆盖范围内,无需单独处理

本次改造的命令执行工具清单限定为:`adb_shell_exec`、`ssh_shell_exec`、`serial_exec` 三个。

### 3. 不处理 PowerShell 通道

PowerShellShell 主要面向本地 Windows 命令,其 exec 模式与常驻命令场景关联不大,本次不纳入。

### 4. 不引入会话空闲超时 / 自动关闭

不新增会话 TTL 或空闲自动清理机制。会话生命周期仍由调用方显式 open/close 管理,与现状一致。

### 5. 不做提示符检测的完美方案

提示符检测只支持常见 prompt 形态并通过配置覆盖,不追求覆盖所有自定义 PS1。对于无法识别提示符的场景,由 maxDuration 熔断兜底,保证不挂死。

### 6. 不新增 GUI / 配置界面

提示符正则覆盖通过现有 config.yaml 的设备配置项实现,不新增任何可视化配置工具。

## 六、 验收标准

### AC1:控制字符工具可发送且语义正确(对应 F1、N5)

连接真实设备打开 adb shell 会话,执行 `logcat` 开始持续输出,调用 `adb_shell_send_ctrl(session_id, key="c")`,命令被 SIGINT 终止,会话恢复可输入状态。抓包或日志验证发送的字节恰好为 `\x03`,无附加换行。

### AC2:exec 前置冲刷消除残留污染(对应 F2)

先用 exec 执行 `logcat` 让其后台持续输出,等待数秒让缓冲区累积数据,再次 exec 执行 `echo hello`,返回内容中只有 `hello` 及提示符,不混入 logcat 的残留日志。

### AC3:常驻命令自动熔断(对应 F3)

打开会话,exec 执行 `logcat`,不指定 maxDuration(走默认 10 秒)。工具在约 10 秒后自动返回,返回内容标注「命令被中断」,且会话恢复可输入状态(再 exec 一条 `echo ok` 能正常返回 `ok`)。

### AC4:瞬时命令不受影响且可提前返回(对应 F1/F3、N1/N3)

exec 执行 `ls /sdcard`,在检测到提示符后立即返回,不再傻等满 delay;返回的文件列表与现状一致。

### AC5:提示符检测 + 配置覆盖(对应 F4)

默认配置下,Android 设备(`:/ $`、`:/ #`)和常见 Linux(`$`、`#`)的提示符能被识别;在 config.yaml 中为某设备配置自定义提示符正则后,该设备的 exec 能正确识别结束。

### AC6:长命令不被误杀(对应 F3、N1)

exec 执行一个耗时约 5 秒的命令(如 `sleep 5; echo done`),不传 maxDuration(默认 10 秒),命令正常完成并返回 `done`,未被熔断。

### AC7:调用方覆盖长命令(对应 F3)

exec 执行一个耗时约 15 秒的命令,显式传 `maxDuration: 30000`,命令正常完成返回,未被熔断。

### AC8:三通道行为一致(对应 F5、N6)

对 adb/ssh/serial 三个通道分别验证:send_ctrl 发送、熔断触发、前置冲刷行为一致,调用方式相同。

### AC9:向后兼容(对应 N1)

现有的 write/read/close 工具签名与行为不变;exec 的既有参数(session_id、command、delay、clear)保留,既有调用方无需改动即可工作。

### AC10:关键事件可观测(对应 N4)

日志中能看到 send_ctrl 的发送、熔断触发(含命令与超时时长)、前置冲刷等事件记录。
