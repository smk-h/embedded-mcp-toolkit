# 串口 ZMODEM 文件传输 Spec

## 背景

`embedded-mcp-toolkit` 通过串口（serial）通道管理嵌入式 Linux 板卡。串口是这类设备（尤其纯串口、无网口的调试场景）唯一的交互通道。当前串口能力仅限于交互式 shell 命令收发，无法传输二进制文件——而固件升级、日志拉取、配置下发等场景普遍需要传 10MB 量级的二进制文件。

### 痛点 1：串口被 MCP 长连接独占

串口采用长连接会话模型：`serial_open` 打开后，该 COM 口被 `portToSession` 映射锁定（`src/mcp/tools/serial/shell.ts`），直到 `serial_close` 才释放。这意味着：

- **外部文件传输程序无法使用该串口**——只要 MCP 会话开着，任何独立进程（如系统自带的 sz/minicom 传输）都拿不到 COM 口
- 若要让外部程序传输，必须先 `serial_close` 让出串口，传完再 `serial_open` 重连——会话中断，可能需重新走 PSH 解锁 / 登录流程，代价大且繁琐

### 痛点 2：现有数据通道是「文本态」，无法承载二进制

现有串口数据通路是面向文本设计的，两处关键代码会把二进制字节篡改：

- **发送侧**：`BaseShell.write(data: string)` 只接受字符串（`src/transports/base-shell.ts`），最终经 `SerialShell.rawWrite(payload: string)` 写出。任何含 `0x80~0xFF` 的字节经字符串编码会失真。
- **接收侧**：`SerialShell` 的数据监听在 `serialPort.on("data", data => this.appendData(data.toString()))` 处，把收到的 `Buffer` 用 UTF-8 `toString()` 强转字符串（`src/transports/serial.ts`），再丢进文本态的 `OutputBuffer`。`0x00`、`0xFF` 等二进制字节经 UTF-8 编解码后被污染。

因此，即使强行把文件 base64 后用 `serial_write` 灌入，或让设备回传二进制再用 `serial_read` 读出，两端解析回来的数据都是错的。**二进制文件传输必须有一条字节透传的通路。**

### 物理瓶颈（预期管理）

串口带宽是物理硬约束，与方案无关：

| 波特率 | 有效字节率 | 传 10MB 耗时（含协议开销） |
|--------|-----------|--------------------------|
| 115200（项目默认） | ~11 KB/s | 约 16 分钟 |
| 921600 | ~90 KB/s | 约 2 分钟 |
| 1.5M~3M（USB 转串口） | 150~300 KB/s | 30~60 秒 |

10MB+ 在 115200 下传输耗时长，属物理瓶颈，非本方案能解决。本方案聚焦"让传输能正确、可靠地进行"，速率由用户选择的波特率决定。

### 设备前提

目标设备为嵌入式 Linux，且**已安装 `lrzsz`**（`rz`/`sz` 命令）。`rz` 是 ZMODEM 协议的接收端实现，`sz` 是发送端实现。这是嵌入式 Linux 几乎标配的工具，设备端无需额外改造。

### 对照：为何不直接走 SSH/SFTP

项目已有成熟的 SFTP 文件传输能力（`src/mcp/tools/ssh/sftp.ts`），10MB 文件走网络是秒级传输，远超串口。但本场景的设备**纯串口、无网络通道**（无网口 / 未开 SSH），SFTP 无法使用，必须走串口。

## 目标

- 在**不释放串口**的前提下，于现有串口会话上实现二进制文件的双向传输（上传到设备 / 从设备下载）
- 传输全程会话保持不断，传输结束后可继续在原会话上执行 shell 命令，无需重连
- 复用设备端已有的 `rz`/`sz`（ZMODEM 协议），MCP 进程内实现 ZMODEM 的对端，无需在设备上部署任何额外程序
- 传输结果以与现有 SFTP 工具一致的摘要格式返回（字节数 / 耗时 / 速率）

## 功能需求

### F1：文件上传工具（serial_upload）

提供一个文件上传能力，将本地二进制文件传输到设备端。行为：

- **输入**：已建立的串口会话 ID、本地源文件路径、（可选）远端文件名 / 远端目录、（可选）设备端接收命令、（可选）超时时长
- **触发设备接收**：通过串口会话向设备 shell 发送接收命令，让设备进入 ZMODEM 接收等待态。默认命令为 `rz`，允许调用方覆盖（如传 `rz -e` 启用控制字符转义）
- **传输角色**：MCP 进程作为 ZMODEM **发送端**，设备端的 `rz` 作为接收端
- **协议**：ZMODEM（含 CRC 校验、分块、流控），由 ZMODEM 协议库实现协议层，MCP 只负责"喂数据 / 收回执"
- **结果**：返回传输摘要（方向、本地路径、远端路径、字节数、耗时、速率，或失败原因）
- **完整文件名传递**：ZMODEM 协议本身会携带文件名，设备端 `rz` 据此命名落盘文件；调用方可指定远端文件名，未指定时取本地文件 basename

### F2：文件下载工具（serial_download）

提供一个文件下载能力，从设备端拉取二进制文件到本地。行为：

- **输入**：已建立的串口会话 ID、远端源文件路径、本地目标文件路径、（可选）设备端发送命令、（可选）超时时长
- **触发设备发送**：通过串口会话向设备 shell 发送发送命令，让设备进入 ZMODEM 发送态。默认命令为 `sz <远端路径>`，允许调用方覆盖（命令中含占位符替换远端路径）
- **传输角色**：MCP 进程作为 ZMODEM **接收端**，设备端的 `sz` 作为发送端
- **协议**：同 F1，ZMODEM 协议库实现接收端逻辑
- **结果**：返回传输摘要（同 F1 结构）；失败时清理半写出的本地文件

### F3：字节透传通路（二进制旁路）

为支撑 ZMODEM 二进制协议帧的正确收发，提供一条绕过现有文本态缓冲区的字节透传通路。行为：

- **发送**：支持以原始字节（而非字符串）写入串口，不追加换行符、不做字符串编码转换
- **接收**：支持挂载一个原始字节接收回调；挂载期间，串口收到的字节在喂给该回调的同时，仍按原样进入文本态缓冲区（不影响 `serial_read` 等现有工具的行为）；卸载后恢复纯文本态
- **隔离性**：该通路是 ZMODEM 等二进制协议的专用通道，默认不启用；不启用时串口行为与改动前完全一致
- **会话内可逆**：挂载与卸载在同一会话内可反复进行，卸载后串口恢复纯文本态

### F4：阻塞式调用 + 进度反馈 + 超时中止

工具调用的执行模型与进度反馈方式：

- **阻塞式**：工具调用在传输完成（或失败、或超时）后才返回，与现有 `serial_exec` 的同步执行风格一致。调用期间进程阻塞在该工具内
- **进度反馈**：传输过程中通过 logger 在 stderr 输出进度信息（已传字节数 / 百分比），不通过工具返回值中途返回进度
- **超时中止**：支持超时参数；超时后主动中止 ZMODEM 会话，返回失败结果并附带已传输字节数；中止后串口会话仍可用
- **超时默认值**：提供合理的默认超时（如 300 秒），覆盖大多数中等文件传输；大文件场景由调用方显式传入更长超时

### F5：设备端命令可覆盖

设备端进入 ZMODEM 收发态的命令可由调用方覆盖，以适配不同设备环境：

- **上传默认**：`rz`
- **下载默认**：`sz {remote}`（`{remote}` 为远端路径占位符）
- **覆盖方式**：调用方通过参数传入完整命令字符串（如 `rz -e`、`sz -e {remote}`）；未传则用默认值
- **目的**：不同 lrzsz 版本、不同设备对转义 / 流控参数的需求不同；允许覆盖避免硬编码导致的兼容性问题

## 非功能需求

### N1：兼容性（不破坏现有串口工具）

- **现有串口工具行为不变**：`serial_open` / `serial_close` / `serial_write` / `serial_read` / `serial_exec` / `serial_send_ctrl` / `serial_shell_login` / `serial_enter_uboot` 的行为与改动前完全一致
- **字节旁路默认不启用**：未挂载字节接收回调时，串口 data 事件的处理路径与改动前逐字一致（仍只走 `appendData(toString)`）；`write(string)` 仍只接受字符串
- **不影响其他通道**：本次改动只涉及串口通道，不改动 SSH / ADB / PowerShell 通道的任何行为

### N2：复用现有摘要格式

- 传输结果摘要（字节数 / 耗时 / 速率格式化）与现有 `ssh_sftp_upload` / `ssh_sftp_download` 的输出风格一致，便于 AI 和用户跨通道理解
- 摘要格式化逻辑（`formatBytes` / `formatRate` / `formatTransferSummary`）从 SSH 通道提取为共享工具，供串口通道复用，避免重复实现

### N3：可观测性

- 传输过程通过 logger 输出关键节点信息：开始传输（本地路径 / 远端路径 / 文件大小）、进度（已传字节 / 百分比，按合理频率）、结束（成功 / 失败、字节数、耗时、速率）
- 失败时记录失败原因（超时 / 协议错误 / 文件不存在 / 设备命令无响应等）
- 进度输出频率适中，避免在 16 分钟的传输中刷屏

### N4：编码规范

- 遵循项目既有的 TypeScript 编码规范（对应 `ts-lang-spec` 技能）
- 新建文件 UTF-8 无 BOM、LF 换行；修改已有文件保持原编码与换行符不变

## 不做的事

- **不改 SSH/SFTP 通道**：SFTP 已成熟，本次不触碰；有网络通道的设备应优先走 SFTP
- **不做断点续传的用户可见恢复 UI**：ZMODEM 协议层支持断点续传（ZRPOS 偏移重传），但本次不为续传提供专门的"继续上次中断"交互；超时中止后不自动续传，由调用方决定重试
- **不引入异步状态查询工具**：不提供 `serial_transfer_status` / `serial_transfer_cancel` 等异步工具；采用阻塞式单次调用，简化设计
- **不做 YMODEM / XMODEM / Kermit 等其他协议**：仅 ZMODEM 一种（lrzsz 原生支持，覆盖绝大多数场景）；U-Boot 的 `loady`/`loadb` 场景不在本次范围（那是 uboot 专用，未来可另开题）
- **不提升串口物理速率**：波特率由设备配置决定，本方案不干预；速率瓶颈是物理层的，非协议层能解决
- **不处理设备端未安装 lrzsz 的情况**：设备端 `rz`/`sz` 缺失时，ZMODEM 无法建链，工具会在超时后返回失败——属预期可诊断行为，不做特殊兜底（如自动降级为别的协议）
- **不持久化传输记录**：不记录历史传输日志到文件（终端日志除外，走现有 fileLogger 机制）；传输摘要只通过工具返回值呈现

## 验收标准

### AC1：上传二进制文件，两端字节一致

上传一个含全字节范围（`0x00`~`0xFF` 均覆盖）的二进制文件到设备，传输完成后在设备端对落盘文件做校验（如 md5sum），与本地源文件校验值一致。

验证：构造一个覆盖 `0x00~0xFF` 的测试文件（如 1KB），`serial_upload` 上传后在设备端 `md5sum <远端路径>`，与本地 `md5sum <本地路径>` 比对一致。

### AC2：下载二进制文件，两端字节一致

从设备下载一个二进制文件到本地，下载完成后对本地文件做校验，与设备端源文件一致。

验证：在设备端准备一个二进制文件（如 `/bin/sh`），`serial_download` 下载后本地 `md5sum` 与设备端 `md5sum` 比对一致。

### AC3：传输全程不释放串口，会话保持可用

从 `serial_upload` / `serial_download` 调用前到调用后，不调用 `serial_open` / `serial_close`；传输完成后，在**同一会话**上执行 `serial_exec "echo ok"` 能正常返回，证明串口未被释放、会话未断。

验证：记录传输前的 session_id，传输后用同一 session_id 调 `serial_exec`，正常返回输出；全程不出现串口被释放的日志。

### AC4：大文件进度反馈

上传 / 下载一个数 MB 的文件，传输过程中 logger 按合理频率输出进度（已传字节 / 百分比），传输结束后返回的摘要含合理的字节数、耗时、速率。

验证：上传 5MB 文件，观察 stderr 出现多次进度日志（如每完成一定字节一次）；最终摘要的 bytes ≈ 5MB、durationMs > 0、rate 与波特率量级匹配。

### AC5：超时中止并返回已传字节

设置较短超时（如 2 秒）上传一个大文件，2 秒后工具返回失败结果，结果中含已传输字节数；中止后串口会话仍可用（同 AC3 验证方式）。

验证：`serial_upload` 传 `timeout=2` 上传 5MB 文件，2 秒内返回 `success: false` 且含 `bytes > 0`；随后同 session 调 `serial_exec` 正常。

### AC6：设备端命令可覆盖

调用 `serial_upload` 时传入自定义接收命令（如 `rz -e`），设备端按该命令进入接收态，传输正常完成。

验证：`serial_upload` 传 `recv_cmd="rz -e"`，传输成功；通过串口日志或设备端确认 `rz -e` 被执行。

### AC7：默认命令下正常工作

不传任何命令覆盖参数，`serial_upload` 默认触发 `rz`、`serial_download` 默认触发 `sz <远端路径>`，均能正常完成传输。

验证：不传 `recv_cmd` / `send_cmd` 调用两个工具，传输成功。

### AC8：现有串口工具行为不变

改动前后，对现有 8 个串口工具（`serial_open` 到 `serial_enter_uboot`）的调用行为与输出完全一致；字节旁路未挂载时，串口 data 事件处理路径与改动前一致。

验证：对照 `src/transports/serial.ts` 和 `src/mcp/tools/serial/shell.ts` 的 git diff，确认现有工具逻辑未被改动；运行现有串口工具观察行为正常。

### AC9：失败场景可诊断

以下失败场景返回明确的失败原因（而非静默挂起或崩溃）：
- 本地文件不存在 → 返回文件不存在的错误
- 设备端命令无响应（如 rz 未安装）→ 超时后返回失败
- ZMODEM 协议错误（如校验失败无法恢复）→ 返回协议错误

验证：分别构造上述三种场景，调用工具，观察返回结果含对应的失败原因描述。

### AC10：传输摘要格式与 SFTP 一致

`serial_upload` / `serial_download` 返回的摘要文本结构与现有 `ssh_sftp_upload` / `ssh_sftp_download` 一致（含 Upload/Download succeeded/failed、local、remote、size、time、rate 字段或等价信息）。

验证：对比两个通道工具的返回文本结构，字段与格式风格一致。
