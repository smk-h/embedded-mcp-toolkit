## 一、 背景

我在 Windows 上跑了一个 claudecode 或者 opencode，启动当前 MCP server，创建一个 SSH 会话连接到编译服务器，现在我想要使用 claude 或者 opencode 分析编译服务器的源代码。

> 已有上下文：[`vscode-sshremote.md`](vscode-sshremote.md) 分析了 VSCode Remote-SSH「无感远程开发」的实现原理——见「把无 UI 内核塞进远程、本地只留壳、用轻量 RPC 经 SSH 隧道交换消息」。本文讨论的是**另一个场景**：AI（claudecode / opencode）作为消费者，怎么拿到远端源代码来做分析。两者可以对照阅读。

## 二、 有哪些思路？

当前 MCP server 已有 SSH 会话工具（`ssh_shell_login` / `ssh_shell_open` + `exec` / `read` / `write` + `close`），但这些工具**只能跑命令、回输出**，并没有「读文件内容到本地」这条直接路径。AI 要分析远端源代码，本质上只有三种思路：

```
思路 A: 把代码搬到 AI 能直接读的地方      (本地优先, 远端只作数据源)
思路 B: 把 AI 的"读/写"能力延伸到远端     (远端优先, 本地只作编排者)
思路 C: 让 AI 在远端就地分析              (远端跑 agent, 本地只发任务)
```

具体到落地，共 6 个主流方案，下面三节按这三类思路展开，最后给出选型建议。

### 1. 思路 A：把代码搬到本地

核心特征：Windows 本地维护一份代码副本，AI 在本地读、本地改，改完再想办法同步回编译服务器。

#### 1.1 方案 A1：批量同步到本地（rsync / scp / tar）

##### 1.1.1 做法

- 在编译服务器把源代码目录暴露为可读；
- 本地通过 `scp -r` / `rsync -avz --exclude=.git` 把整树拉到 Windows 本地工作区（如 `E:\AI\remote-src\<server>`），或者用 `tar czf - <dir>` 打包后传输、本地解压；
- 让 claudecode / opencode 直接打开本地工作区，AI 用本地 Read / Grep / Glob 工具分析，零改动。

```
+-------------------+   rsync/scp/tar  +------------------------+
| 编译服务器          | =============> | Windows 本地工作区       |
| ~/src/kernel/...   |                | E:\AI\remote-src\...   |
+-------------------+                +------------------------+
        ^                                      ^
        |                                      | 本地 Read/Grep/Glob
        |                                      v
        \----------------------------- claudecode / opencode
```

##### 1.1.2 优点

- 实现最快，不依赖 MCP server 的文件类新工具；
- AI 用现成的本地工具链（Read、Grep、Glob），上下文窗口里直接放文件内容、片段、行号引用全部精确；
- 离线后仍能继续分析，可多次反复 grep；
- 与 git 工作流衔接自然（拉回的就是可 commit 的本地仓库）。

##### 1.1.3 缺点

- 需要本地磁盘空间（大仓库可能 GB 级）；
- 远端持续修改时存在「快照陈旧」问题——需要每次开新一轮分析前手动再同步一次；
- 拉取过程本身有带宽成本，首次同步较慢；
- `rsync` 适合增量同步（基于内容差异的 diff 传输，是 A1 相对 B2 的隐性优势），`tar` 适合一次性大目录快照，两者都需要用户判断用哪个。

##### 1.1.4 写回

**支持，但需要额外同步步骤。**

AI 改完本地文件后，需要通过 `rsync` / `scp` 把变更推回编译服务器。这不是自动的，必须由 AI 显式调用同步工具，或由用户在编译前手动同步。如果忘记同步，编译的仍然是远端旧代码。

##### 1.1.5 适用

- 远端源码总体稳定、不频繁变动；
- AI 需要做跨文件、反复 grep、整体结构理解这类**重读**任务；
- 本地磁盘和带宽都够用；
- 可以接受「本地改完再推回去」的两步工作流。

#### 1.2 方案 A2：子目录 / 单文件按需拉取

##### 1.2.1 做法

- 不全量拉，而是让 AI 通过 `ssh_shell_exec` 执行 `cat` / `head` / `tail` / `sed -n 'a,bp'` 之类的命令，按需把**当前要分析的那一段**内容取回（注意：这是「按行号区间读」，需 AI 自己算行号；与「按内容差异 diff 读」心智模型不同，后者要求远端维护 baseline）；
- 输出会出现在 `ssh_shell_exec` 的返回值里，AI 拿到字符串后即可在上下文中推理。

```
AI -> ssh_shell_exec(session, "sed -n '120,180p' drivers/net/e1000.c")
       |
       v
SSH 通道 -> 远端 sed -> 返回那段文本
       |
       v
AI 上下文里得到那段代码 -> 继续推理
```

##### 1.2.2 优点

- **零本地存储**，磁盘压力小；
- **零部署**，完全用现有 MCP server 的 `ssh_shell_exec` 工具就能干；
- 每次只取真正相关的那一段，上下文窗口利用率高。

##### 1.2.3 缺点

- 每个文件都要「先想读哪段 → 发命令 → 等回显」三步往返，延迟累加；
- 命令行输出会被终端宽度截断、被转义字符污染（颜色码 / `\r`）——需要在 MCP server 侧或 AI 侧做净化；
- 没有全局视图，AI 难以知道仓库全貌，得靠 `ls` / `find` 边走边探；
- 长会话容易触发 SSH 输出缓冲的截断（受 `MAX_BUFFER_SIZE` 约束，会在大文件 cat 时被切）。

##### 1.2.4 写回

**不支持，或极不推荐。**

`ssh_shell_exec` 本质上是跑命令，AI 要改文件只能再发一条 `sed -i` 或 `echo > file` 命令，风险高、不可预测、没有回滚能力。因此该方案只适合只读分析，不适合 ch02 的「AI 改代码 → 编译」闭环。

##### 1.2.5 适用

- 已经定位到目标文件、只想深入读几段，不需要全局漫游；
- 仓库极大、本地不便整拉；
- 临时排查 bug、看少数关键函数；
- **只读场景**，不需要 AI 修改代码。

### 2. 思路 B：把 AI 的读写能力延伸到远端

核心特征：代码仍然留在编译服务器上，AI 通过 MCP tools 直接读写远端文件，本地不维护或只维护临时副本。

#### 2.1 方案 B1：远端索引与搜索查询（cscope / ripgrep）— 只读辅助

##### 2.1.1 做法

- 在编译服务器上预跑 `cscope -b -R`、`ctags -R`、`universal-ctags` 建索引；
- AI 通过 `ssh_shell_exec` 调用 `cscope -L1 <symbol>` / `rg <pattern>` / `grep -rn` 来查「谁调用了我」「定义在哪」「这个宏被哪些文件用」；
- 只回传结构化的查询结果，不传源代码正文。

```
AI -> ssh_shell_exec: "cscope -L1 e1000_transfer"
       |
       v
远端 cscope 查询本地索引
       |
       v
返回 "(file.c:line) call site 1\n..."
       |
       v
AI 在上下文里拿到调用关系, 再按需 sed -n 读具体行
```

##### 2.1.2 优点

- **查询本身极快**，远端 cscope 索引在内存里，调 k 行结果秒回；
- 网络流量极小，只传查询结果不传源码；
- 适合做大型代码库的「跨文件符号关系」分析，补 A2 的短板。

##### 2.1.3 缺点

- 依赖编译服务器有 `cscope/ctags/ripgrep`，且索引要预先建好（首次较慢）；
- **索引陈旧**：远端源码改动后要重新 `cscope -b`，否则结果不匹配；
- AI 只能拿查询结果，真要看代码片段仍需 A2/B2 配合，工作流是「查索引 → 读文件」两段。

##### 2.1.4 写回

**不支持。**

B1 只能搜索和定位，本身不提供文件修改能力。它可以作为 B2/C1 的辅助（先搜索再修改），但不能独立完成「改代码 → 编译」闭环。

##### 2.1.5 适用

- 仓库巨大（如整个 Linux 内核、整个 Android 框架），不可能整拉到本地；
- AI 任务以「定位、调用链、依赖分析」为主；
- 编译服务器本身就是开发主战场，索引长期维护；
- 配合 B2 / C1 使用，先搜索再修改。

#### 2.2 方案 B2：MCP 文件工具层（推荐演进方向）

##### 2.2.1 做法

给当前 MCP server 新增一组语义化文件工具，把 `ssh_shell_exec` + 一堆 shell 命令的细节封装成统一的「读写远端文件」接口：

| 新工具名 | 等价远端操作 | 作用 |
|---------|------------|------|
| `ssh_file_read` | `cat` / `sed -n 'a,bp'` / SFTP readFile | 按路径 + 可选行号区间读远端文件（局部读，非内容 diff） |
| `ssh_file_write` | SFTP writeFile / 原子写 | **全量覆盖写**——用于新建文件或整文件重写 |
| `ssh_file_edit` | SFTP 局部写 + 行号/oldStr 匹配 | **局部 patch 写**——ch02「AI 改代码」主力写工具，详见 2.2.4 |
| `ssh_file_list` | `find` / `ls -la` / SFTP readdir | 列目录树 |
| `ssh_file_grep` | `rg --json` / `grep -rn` | 远端搜索，回传结构化匹配 |
| `ssh_file_stat` | `stat` / SFTP stat | 拿 mtime / size，用于判断是否变化 |

底层可以有两种实现方式：

###### 方式一：shell 命令翻译

AI 给 path + lineRange，MCP server 翻译成 `sed -n 'a,bp'` 等命令执行，再净化输出。实现快，但受终端输出、ANSI 码、`MAX_BUFFER_SIZE` 限制。

###### 方式二：SFTP 结构化访问

MCP server 通过 `ssh2` 打开 SFTP 子通道，直接调用 `readFile` / `writeFile` / `readdir` / `stat` 等原语。文件内容以二进制安全方式传输，不走 pty/shell，输出干净、稳定。

```
AI -> ssh_file_read(path="drivers/net/e1000.c", start=120, end=180)
       |
       v
MCP server 内部:
  ├─ shell 方式: ssh exec "sed -n '120,180p' drivers/net/e1000.c"
  └─ SFTP 方式: sftp.readFile("drivers/net/e1000.c") 后按行切片
       |
       v
返回干净文本(带行号)
       |
       v
AI 上下文塞进去继续推理
```

##### 2.2.2 优点

- AI 不需要懂 shell 命令细节，**调用面干净**——传 path + lineRange 即可；
- 输出经 MCP server 统一净化（去 ANSI、合并 `\r\n`），避免 A2 的转义污染陷阱；
- 可以做分块流式返回（大文件分块），缓解 `MAX_BUFFER_SIZE` 截断问题；
- SFTP 方式复用 SSH 连接、协议标准、单文件读写稳定，比 sshfs 性能好；
- 同样的接口形态可以**并行**支持本地（直接 fs）/ 远程/容器多种后端，对 AI 一致。

##### 2.2.3 缺点

- **需要开发**，不是零成本方案；
- 工具粒度与命令性能之间存在权衡（要不要支持 glob？要不要支持 follow symlink？）；
- 文件级状态（mtime 缓存等）需要会话级管理；
- SFTP 方式大量小文件随机访问时仍有 RTT 累积，批量同步不如 rsync。

##### 2.2.4 写回

**支持，直接写回远端。** 写回分两种粒度，需按场景选择：

| 写工具 | 粒度 | 适用场景 | 风险/约束 |
|--------|------|---------|----------|
| `ssh_file_write` | 全量覆盖 | 新建文件、整文件重写 | AI 易在重写时漏行；无 baseline 校验，会静默覆盖远端并发改动 |
| `ssh_file_edit` | 局部 patch | AI 修改大文件中的几处（ch02 主力） | 省 token；可带 baseline（oldStr / 行号快照）做乐观锁，远端已变则拒绝写 |

- **全量写 vs patch 写的权衡**：几千行的 C 源文件，全量读全量写每次都吃掉大量上下文窗口，AI 还容易丢内容；patch 写只需保证片段正确，且能通过 baseline 校验防止「AI 读 t1 快照、远端已到 t2」的静默覆盖问题。
- `ssh_file_edit` 的参数建议为 `path + (lineRange | oldStr) + newContent + optional baseline`，语义对齐本地 edit 工具（`old_str→new_str`）。

##### 2.2.5 适用

- 长期、反复用 AI 分析并修改远端源码；
- 希望一次开发、多种 transport（SSH / ADB / Serial）共享同一文件抽象；
- 是 ch02 最值得推进的方向之一。

##### 2.2.6 可扩展的高级工具

B2 的抽象层之上，未来还可以叠加一些高级工具，但它们不是独立方案，只是 B2 工具集的扩展：

| 高级工具 | 作用 | 与方案的关系 |
|---------|------|------------|
| `ssh_file_summarize` | 对单个文件或目录生成摘要（函数签名、关键注释、依赖列表） | 原作为独立摘要方案的能力，现合并为 B2 的一个 tool |
| `ssh_file_index` | 调用远端 `cscope` / `ctags` 建索引，或维护项目级符号索引 | 替代/增强原「B1 远端索引查询」，AI 先搜索引再读文件 |
| `ssh_file_diff` | 比较远端文件与本地/历史版本差异（**只读比较**，不是 patch 写回） | 辅助 AI 判断修改范围，与 `ssh_file_edit` 的写操作区分 |

这些工具可以按需实现，不影响 B2 的核心架构。

#### 2.3 方案 B3：远端文件系统挂载（sshfs / rclone）

##### 2.3.1 做法

- 在 Windows 上用 `sshfs-win` / `rclone mount` / WinSCP 挂载编译服务器的源码目录到本地盘符（如 `Z:\remote-src\`）；
- 让 claudecode / opencode 直接像访问本地文件一样 Read / Grep / Write 它。

##### 2.3.2 优点

- 对 AI **完全透明**——它根本不知道文件在远端，工具链零改动；
- 实时反映远端改动（挂载语义是「按需读」），不会像 A1 那样陈旧；
- 不占本地磁盘空间（按需 fetch）；
- 写操作自动同步回远端。

##### 2.3.3 缺点

- Windows 上 `sshfs-win` / WinFsp 稳定性一般，长会话可能掉盘；
- 每次 Read 都要触发一次 SFTP 网络往返，**搜索（grep）极其痛苦**——每次匹配都是一次网络 IO，远比 A1 慢；
- 不适合 AI 反复读同一文件、来回 grep 的工作模式（延迟累计）。

##### 2.3.4 写回

**支持，自动写回。**

AI 修改本地挂载盘中的文件，底层会自动通过 SFTP 写回编译服务器。对 AI 来说和改本地文件没有区别。

##### 2.3.5 适用

- 文件量小、读次数少、要严格保持实时；
- 不是 AI 重度分析场景，AI 只偶发看/改一两个文件——但偶发场景用 A2/B2 更轻。

##### 2.3.6 不推荐用于 AI 重分析

原因即上述搜索性能问题，正是 [`vscode-sshremote.md`](vscode-sshremote.md) 第八节对比里讨论过的「挂载网络盘」痛点。

### 3. 思路 C：远端跑一个服务

核心特征：在编译服务器上跑一个完整或精简的服务，AI 通过网络协议远程接入，所有操作在服务器本地完成。

#### 3.1 方案 C1：远端部署 MCP server（JSON-RPC 远程服务进程）

##### 3.1.1 做法

- 把现在的 MCP server 直接跑到编译服务器上（Linux 版本），以 stdio 方式运行；
- 本地 claudecode / opencode 通过 SSH exec 建立 stdin/stdout 的 JSON-RPC 通道远端接入（不开远程端口），所有文件工具直接命中远端本地文件系统——**就是 VSCode Remote-SSH 的「反向」版本**。

```
+---------- Windows ----------+                      +----- 编译服务器 -----+
| claudecode/opencode          | MCP over SSH/JSON-RPC | MCP server (Linux)    |
|   |  远端 MCP 客户端         | <==================> | file tools (本地 fs)   |
+-----------------------------+                      +------------------------+
                                                              |
                                                              v
                                                       [远端本地磁盘]
                                                       [远端 cscope / rg]
                                                       [远端 build / git]
```

##### 3.1.2 优点

- 体验最接近 VSCode Remote-SSH 的「无感」——文件读写、grep、构建都在远端就地完成；
- AI 看到的就是远端**实时**状态，无任何同步开销；
- 与 ch02 长远目标「让远端 AI 流程能反向访问本地设备」天然对齐——可在远端 MCP server 上再加「本地设备代理」工具把本地 USB /串口反向暴露出去。

##### 3.1.3 缺点

- 需要把 MCP server 移植 / 部署到 Linux，要解决跨平台编译、Node 依赖、IPC 文件路径差异；
- 鉴权依托 SSH 账号体系（走 SSH exec 通道不开远程端口，比裸跑 TCP/SSE 安全），但仍需注意 SSH 账号权限收敛；
- 仍然是「AI 不会把源码搬到本地」，**远端跑挂 = AI 失明**，没有离线能力；
- 当前 MCP server 架构是面向本地（Windows 主机）的，移到远端后 PowerShell 工具、serial 工具、port_scan 等就失去意义，要做裁剪。

##### 3.1.4 写回

**支持，直接写回远端。**

AI 调用远端 MCP server 的文件 tools 修改文件，修改直接发生在编译服务器本地。这是最强的写回之一。

##### 3.1.5 适用

- 长期、合作式开发场景——编译服务器是主战场，AI 反复分析与修改同一仓库；
- 已经决定把 AI 工作流从「本地控制远端」**演进**为「远端就是 AI 工作空间」——这是 ch02 终局方向之一。

### 4. 方案对比

前文在思路 A/B/C 下共讨论了 6 个细分方案，但真正适合 ch02「AI 改代码 → 远程编译 → 本地烧录」闭环的，其实只有三种落地形态。本节先给出六类细分方案的速览，再重点对比三种可落地方案，便于选型。

#### 4.1 六类细分方案速览

| 方案                 | 本地副本 | 透明度 | 实时性 | 搜索性能 | 读写粒度               | 开发成本 | 写回     | 离线可用 |
| :------------------- | :--- | :----- | :----- | :------- | :--------------------- | :------- | :------- | :------- |
| A1 批量同步到本地    | 大   | 高     | 快照   | 极快     | 读局部/写局部/同步增量 | 零       | 需同步   | 可以     |
| A2 按需 sed 读       | 无   | 低     | 实时   | 慢(单点) | 读局部(按行号)         | 零       | 不支持   | 不可     |
| B1 远端索引查询      | 无   | 中     | 实时   | 极快     | 不读正文               | 小       | 不支持   | 不可     |
| B2 MCP 文件工具层    | 无   | 高     | 实时   | 中       | 读局部/写全量+patch    | 中       | 直接写回 | 不可     |
| B3 sshfs 挂载        | 微   | 极高   | 实时   | 痛苦     | 读局部/写局部          | 小       | 自动写回 | 不可     |
| C1 远端跑 MCP server | 无   | 极高   | 实时   | 极快     | 读局部/写局部或全量    | 大       | 直接写回 | 不可     |

- **只读方案**：B1（远端索引查询）——只能帮 AI 分析，不能改代码，必须配合 B2 / C1 使用。
- **读写但需本地副本 + 同步**：A1（批量同步到本地）——AI 改本地，改完要推回去，存在两步工作流和版本一致性问题。
- **直接读写远端**：B2（MCP 文件工具层）、B3（sshfs 挂载）、C1（远端跑 MCP server）——AI 的修改直接命中编译服务器，是 ch02 主力方向。
- **不推荐写回**：A2（按需 sed 读）——只能读片段，写回风险高、不可控。
- **摘要不是独立方案**：文件摘要应作为 B2 / C1 工具层里的一个高级 tool（如 `ssh_file_summarize`），与 B1 的索引查询类似，属于辅助能力。

最终可以选择 JSON-RPC 远程服务进程，是因为它不仅解决了「读写远端文件」，还把「搜索、编译、状态查询、产物下载」统一到一个远程执行环境中，是三种方案中唯一能够支撑完整自动化闭环的架构。

## 三、 结论

- **只读分析**：B1（远端索引查询）是有效辅助，但不能独立完成改代码闭环；文件摘要应作为 B2 / C1 工具层里的一个高级 tool 实现，不是独立方案。
- **短期最实用**：A1（rsync 拉本地）+ B1（远端索引辅助），零开发，30 分钟可用；缺点是改完代码后需要额外同步步骤。
- **中期最值得做**：B2（MCP server 增设文件工具层，优先 SFTP 实现）——AI 直接读写远端文件，无本地副本，无同步步骤，是 ch02 主力方向之一。注意读写粒度：`ssh_file_read` 走局部读；写回需同时提供 `ssh_file_write`（全量覆盖）和 `ssh_file_edit`（局部 patch + baseline 校验）两种粒度，后者是 AI 改大文件的主力工具。
- **三种可落地方案**：JSON-RPC 远程服务进程（统一、可扩展）、B2 + SFTP（稳健折中）、sshfs 挂载（最小开发量但稳定性差）。
- **远期终局**：C1（远端跑 MCP server，本地 AI 远程接入），与 ch02 「跨边界设备访问」终极目标对齐。

落地节奏建议：**先把 A1 + B1 跑通验证 AI 分析质量，再投入 B2 或 JSON-RPC 实现远程直接读写**——避免过早抽象。
