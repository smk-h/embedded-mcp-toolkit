# 日志文件命名与目录结构优化 Spec

## 背景

当前 MCP 服务产出两类日志文件，均落在 `LOG_DIR` / `SAVE2FILE_PATH` 指向的同一目录（默认 `.embedded/log/`）：

1. **MCP 命令操作日志** —— 进程级单例，由 `logger.ts` 写入，文件名格式 `YYYY-MM-DD_HH-MM-SS.log`（如 `2026-07-18_13-54-00.log`）。
2. **终端原始日志** —— 每个 shell 会话一份，由 `file-logger.ts` 写入，文件名格式 `<sessionId>_YYYY-MM-DD_HH-MM-SS.log`（如 `serial_1_2026-07-18_13-53-01.log`）。其中 `sessionId` 形如 `serial_1`、`ssh_2`、`power_1`，是 `<通道前缀>_<自增计数器>`，**不含设备维度**。

存在两个问题：

- **文件名冗长**：时间戳内部的连字符/下划线（`-` / `_`）拉长了文件名，且与 sessionId 的下划线混在一起，可读性差。
- **终端原始日志无法区分设备**：一个目录里混着所有设备的 `serial_1`、`ssh_2`，多设备调试时无法快速定位某台设备的会话日志。

## 目标

- 压缩两类日志文件名中的时间戳，去除时分秒之间的分隔符，采用 `YYYY-MM-DD_HHMMSS` 格式（日期用 `-` 分隔易读，时间紧凑，与 sessionId 之间用 `_` 分隔边界清晰）。
- 终端原始日志按设备名分目录存放，使每台设备的会话日志独立聚合；MCP 命令日志作为进程全局日志，保持在 log 根目录。

## 功能需求

### F1: MCP 命令日志文件名紧凑化
- MCP 命令操作日志文件名中的时间戳格式由 `YYYY-MM-DD_HH-MM-SS` 改为 `YYYY-MM-DD_HHMMSS`，去除时分秒之间的分隔符。
- 文件名整体形如 `2026-07-18_135400.log`。
- 日志仍落在 `LOG_DIR` 指向的目录根下（默认 `.embedded/log/`），**不**按设备分目录（命令日志是进程级全局日志）。
- 文件内部已有的行为保持不变：日志头部的北京时间戳、每行前缀的行内时间戳 `[YYYY-MM-DD HH:mm:ss]` 不受影响。

### F2: 终端原始日志按设备分目录
- 终端原始日志的落盘路径由 `<log_dir>/<sessionId>_YYYY-MM-DD_HH-MM-SS.log` 改为 `<log_dir>/<deviceName>/<sessionId>_YYYY-MM-DD_HHMMSS.log`。
- `deviceName` 取 `resolveDeviceName()` 返回的原值（如 `board-lubancat`），**不**做前缀剥离。
- 设备子目录在写日志前按需自动创建（目录不存在时递归建目录）。
- 一个进程内的所有 shell 会话（adb / serial / ssh / powershell）共享同一个设备子目录，即同一进程产生的所有终端原始日志都落在同一个 `<deviceName>/` 下。

### F3: 文件名时间戳统一
- F1 与 F2 的时间戳格式统一为 `YYYY-MM-DD_HHMMSS`（仅命令日志和终端原始日志的**文件名**受影响）。
- 日志**文件内部**的时间戳格式（头部、行内前缀）保持原样不变。

### F4: 设备名透传
- 终端原始日志启用时（`enableFromEnv`）需要拿到当前设备名；各 shell open handler 在调用 `enableFromEnv` 时透传已解析的 `deviceName`。
- 仅当 `deviceName` 可用时才分目录写日志；若 `deviceName` 解析失败（理论上不会发生，因为 `resolveDeviceName` 有兜底 `board-a`），回退到原 `<log_dir>/` 根目录，保证日志不丢。

## 非功能需求

### N1: 向后兼容性
- 本次改动**只影响新产生的日志文件名和路径**。已有的历史日志文件（旧格式文件名、旧目录结构）**不**做迁移、**不**做清理，由用户自行处理。
- 环境变量 `LOG_SAVE`、`LOG_DIR`、`SAVE2FILE_PATH` 的**语义和默认值不变**，仍按现有规则控制日志启停和目录。

### N2: 排序稳定性
- 新文件名中的时间戳仍保持「年→月→日→时→分→秒」从大到小、定宽排列，确保**按文件名字典序排序等同于按时间排序**，与旧行为一致，不影响任何依赖文件名排序的工具或脚本。

### N3: 平台兼容性
- 设备子目录的创建使用递归建目录能力，目录分隔符使用平台无关的路径拼接（不硬编码 `\` 或 `/`），在 Windows 和类 Unix 环境下行为一致。

### N4: 零运行时风险
- 设备子目录创建失败不应导致会话打开失败或主流程中断——日志写入是辅助能力，出错时降级处理（记录一条错误日志到命令日志，继续运行），不抛出未捕获异常。

## 不做的事

- **不做历史日志迁移**：不对 `.embedded/log/` 下已有的旧格式日志文件重命名或搬迁，旧文件保留原样。
- **不做 MCP 命令日志的分目录**：命令日志是进程级全局日志，本次**不**将其按设备分目录；只终端原始日志分目录。
- **不做设备名前缀剥离**：目录名直接用 `resolveDeviceName()` 原值（如 `board-lubancat`），**不**剥 `board-` 前缀。
- **不改日志文件内部格式**：文件头部、行内时间戳前缀 `[YYYY-MM-DD HH:mm:ss]`、BOM、写入策略等**保持原样**，本次只改文件名/路径。
- **不改 `sessionId` 生成规则**：`<prefix>_<counter>` 格式不变（如 `serial_1`），只是把它放进对应设备子目录。
- **不改环境变量**：`LOG_SAVE` / `LOG_DIR` / `SAVE2FILE_PATH` 的名称、语义、默认值均不变。
- **不引入新依赖**：不引入新的日志库或时间库，复用现有 `timestamp.ts`。

## 验收标准

### AC1: 命令日志文件名格式正确（对应 F1、F3）
- 启用日志（`LOG_SAVE=1`）启动 MCP server 后，`LOG_DIR` 根目录下生成的命令日志文件名形如 `2026-07-18_135400.log`，即日期用 `-` 分隔、时间紧凑无分隔符。

### AC2: 终端原始日志落盘到设备子目录（对应 F2）
- 打开任意通道（adb / serial / ssh / powershell）的 shell 会话后，终端原始日志文件落在 `<LOG_DIR>/<deviceName>/` 下，文件名形如 `serial_1_2026-07-18_135301.log`。
- `deviceName` 与 `resolveDeviceName()` 返回值一致（如 `board-lubancat`）。

### AC3: 设备子目录自动创建（对应 F2）
- 设备子目录不存在时，首次写终端原始日志能自动创建该子目录，且不会报错；多次会话复用同一子目录。

### AC4: 文件名时间戳统一（对应 F3）
- 两类日志文件名的时间戳部分格式一致，均为 `YYYY-MM-DD_HHMMSS`（仅文件名层面）。

### AC5: 日志文件内部格式不变（对应 F1、F3）
- 命令日志的文件头部仍为 `=~=~=~=~=~=~=~=~=~=~=~= Mcp Server log YYYY.MM.DD HH:mm:ss =~=~=~=~=~=~=~=~=~=~=~=`；每行前缀仍为 `[YYYY-MM-DD HH:mm:ss]`。
- 终端原始日志的行内前缀仍为 `[YYYY-MM-DD HH:mm:ss]`。

### AC6: 文件名字典序等于时间序（对应 N2）
- 在同一目录下连续产生多份日志文件，按文件名字典序排序的结果与按文件创建时间排序的结果一致。

### AC7: 历史日志不受影响（对应 N1）
- `.embedded/log/` 下已有的旧格式日志文件（如 `2026-07-18_13-54-00.log`、`serial_1_2026-07-18_13-53-01.log`）在改动后仍保留原文件名、原位置，不被重命名或搬迁。

### AC8: 环境变量语义不变（对应 N1）
- `LOG_SAVE=0` 或未设时不写命令日志；`SAVE2FILE_PATH=none` 或未设时不写终端原始日志；行为与改动前一致。

### AC9: 降级行为（对应 N4）
- 设备子目录创建失败时，会话打开主流程不中断；降级路径在命令日志中留下一条错误记录。
